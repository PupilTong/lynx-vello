# Media & resource APIs: image loading, clipboard, video/audio

> Research: multi-agent sweep over `lynx/` and `lynx-stack/` (see [AGENTS.md](../../AGENTS.md) for the reference-repo shorthand and the W3C-first standards policy). Supersedes the earlier stub.

### Clipboard Interaction and Native Video/Audio Playback

Both items in this gap turn out to be **native-only, platform-layer features that have no web-platform counterpart** — they are not part of the shared C++ `core/renderer` DOM/TASM engine at all, but are registered per-platform as `LynxBehavior`/element plugins (Android Kotlin, iOS Obj-C). This matches Lynx's plugin-style "element/behavior" model where the core engine is agnostic to which tag names exist; `web-core`/`web-elements` (browser-based) simply never implemented equivalents, and grep across `core/renderer`, `core/renderer/dom`, `core/renderer/tasm` confirms zero references to `video` or `clipboard` in the shared engine.

**Video** (`<video>` tag, package `lynx_xelement_video`) is a full native UI component on both platforms: `platform/android/lynx_xelement/lynx_xelement_video/.../LynxUIVideo.kt` (`@LynxBehavior(tagName = ["video"])`, 522 lines) and `platform/darwin/ios/lynx_xelement/video/LynxUIVideo.m` (438 lines) implement an identical prop/method/event surface — confirmed by reading both files side-by-side. It is analogous to `x-audio-tt` (`lynx-stack/packages/web-platform/web-elements/src/elements/XAudioTT/`), which likewise has no video sibling.

**Clipboard** support in Lynx is copy-only and implicit, not a JS-exposed API: long-press-to-select on `<text>` triggers a native "Copy" system menu item that writes to the OS clipboard (Android `ClipboardManager`/`ILynxSystemInvokeService.setPrimaryClip`, iOS `UIPasteboard`/`LynxServiceSystemInvokeProtocol`). No Lepus/JS-facing clipboard module, no programmatic read-clipboard API, and no "paste" or "cut" menu item were found anywhere in `core/`, `platform/android`, or `platform/darwin` (iOS `canPerformAction:` explicitly whitelists only `selectAll:`/`copy:`). `x-input`/`x-textarea` get OS-native cut/copy/paste for free because they wrap real platform text-editing widgets (`AppCompatEditText` on Android; presumably `UITextField`/`UITextView` on iOS) — Lynx does not implement clipboard handling for them itself. There is also a separate, generic `GetClipboardData`/`SetClipboardData` hook in the headless/windowless embedder renderer (`platform/embedder/public/lynx_windowless_renderer.h`), used for non-mobile embedding scenarios (e.g. testing/CI rendering), unrelated to the text-selection copy feature.

**W3C divergence flags:**
- `<video>` is not `HTMLMediaElement`-compatible in any respect (custom queued-command methods instead of direct property/method access, custom event names, no `<source>`/track children, no promise-returning `play()`). This is an intentional Lynx design choice, not a bug — lynx-vello should replicate **Lynx's** API surface for ReactLynx compatibility, not the W3C `<video>` API.
- Clipboard "copy" is implemented as an ad-hoc native text-selection menu action, not the W3C Clipboard API / `ClipboardEvent`s (`copy`/`cut`/`paste` events with `clipboardData`, async `navigator.clipboard.writeText/readText`). Since Lynx itself never implements the W3C model here, there is no "Lynx quirk vs. W3C standard" conflict to resolve in the way z-index is — this is simply a much narrower feature (system copy-menu action only) that lynx-vello should match at the same narrow scope, while being free to layer a fuller OS-clipboard integration under the hood as an implementation detail (not exposed to scripts) if useful for text-input paste support.

### Image Resource Loading Pipeline

Native Lynx's `<image>` (`x-image`/`XImage` in web-platform) does **not** route through the generic `LynxResourceLoader`/`clay/net/loader` machinery examined first (`lynx/core/public/lynx_resource_loader.h`, `lynx/core/resource/lynx_resource_loader_android.cc`, `lynx/core/resource/lynx_resource_loader_darwin.mm`). That path only ever handles `kAssets`/`kExternalJs`/`kLazyBundle`/`kFrame`/`kExternalByteCode` resource types — `LynxResourceType::kImage` is defined in `lynx/core/public/lynx_resource_loader.h` but `LynxResourceLoaderDarwin::LoadResourceInternalSync` (`lynx/core/resource/lynx_resource_loader_darwin.mm:280-356`) has no case for it and falls through to `"Unsupported type."`. Likewise `clay/net/loader/*` (`resource_loader.h`, `resource_loader_factory.cc`, `data_image_loader.cc`) is a separate, older Skia-era abstraction (only handles `data:` URIs and a generic byte-loader interface) that is not what backs the `<image>` element's `src`/`placeholder` in the current Android/iOS UI stack. Image loading is instead a **self-contained per-platform subsystem**: `LynxImageManager`/`LynxImageLoader`/`ILynxImageService` (Android, Fresco-backed) and `LynxUIImage`/`LynxImageManager`/`LynxImageLoader` (iOS, its own `LynxURL`/image-service abstraction), both ultimately delegating to an embedder-registered fetcher (`LynxMediaResourceFetcher` / `LynxGenericResourceFetcher` on Android, `LynxImageFetcher`/generic fetcher on iOS) with a default in-house implementation (Fresco on Android; no default iOS implementation was located, host app must supply `imageFetcher`/`genericResourceFetcher`). web-platform (`lynx-stack/packages/web-platform/web-elements/src/elements/XImage/*`) sidesteps all of this by delegating entirely to the browser's native `<img>` element — no custom Rust/TS resource loader, decode thread, or cache exists in `web-core`; HTTP fetch, HTTP caching, and image decode are 100% the browser's job. This is the biggest architectural gap for lynx-vello: with no browser underneath, lynx-vello must build the equivalent of Fresco/UIImage's fetch+decode+memory-cache+disk-cache pipeline itself (a natural fit for a Rust `image`/`zune-image`-based decoder plus an httpc + LRU cache, decoding off the render thread and handing vello-renderable textures back).

Key architectural facts (native): request lifecycle is driven by a dirty-flag diff (`SRC_CHANGED`, `PLACEHOLDER_CHANGED`, `MODE_CHANGED`, `BLUR_RADIUS_CHANGED`, `AUTO_SIZE_CHANGED`, `DOWN_SAMPLING_SCALE_CHANGED`, `REGION_TO_DECODE_CHANGED`, etc., `lynx/platform/android/.../LynxImageManager.java:81-107`) so a prop update or layout-size change re-issues a fetch only for the changed aspect. `src` and `placeholder` are **two independent, concurrently-issued requests** with independent listeners (`mSrcLoadListenerInner` / `mPlaceHolderListener` on Android; `requestImage:_src` + `requestImage:_placeholder` fired together on iOS, `lynx/platform/darwin/ios/lynx/ui/image/LynxUIImage.mm:654-658`) — there is no "try src, fall back to placeholder on error" logic in native; placeholder is purely an interim visual shown until `src` succeeds, and once `src` succeeds any later/racing placeholder result is discarded (`onSuccess` checks `requestInfo.getUrl() == mCurImageRequest.getUrl()` on Android; on iOS the completion block explicitly returns early for a placeholder result once `strongSelf.image` is already set — `LynxUIImage.mm:812-817`, and `LynxImageManager.mm:79-82` on the newer iOS image-service path literally states "If source image is loaded, placeholder image is not needed to be displayed"). **No automatic retry policy exists anywhere in native** (`ImageErrorCodeUtils.java`, `ImageUrlRedirectUtils.java`, `LynxImageService.java`, all iOS image files were grepped for "retry" — the only "retry" hits are `AutoSizeImage`'s unrelated `mNeedRetryAutoSize` layout re-measure flag); a failed load surfaces once as an `error` event and nothing else re-fires it unless the app resets `src`.

`auto-size` triggers a genuine **post-decode relayout**: `AutoSizeImage.measure()` (`lynx/platform/android/.../image/AutoSizeImage.java`) returns the *previous/placeholder* measured size until the real bitmap dimensions are known, then `justSizeIfNeeded()` calls `markDirty()`/relayout once decode completes and the intrinsic aspect ratio differs meaningfully (>0.05) from the current box — i.e. native Lynx has the same "layout jank on late-arriving intrinsic size" behavior as an `<img>` without explicit `width`/`height`, and a bitmap-size cache (`ILynxViewRuntimeCacheManager.setBitmapSizeCache`) is used to avoid the jank on repeat mounts of the same URL. web-platform sidesteps this entirely by keeping `auto-size` a pure-CSS affair (`x-image.css:55-81`: `display:contents` + `width/height: inherit`/`max-width/max-height:100%` on the inner `<img>`), relying on the browser's native intrinsic-size layout — no JS remeasurement needed.

`mode` (object-fit) is a direct, already-standards-aligned mapping in all three implementations: Android `ScalingUtils.ScaleType` (`FIT_XY`/`FIT_CENTER`/`CENTER_CROP`/`CENTER`, `lynx/platform/android/.../image/ScalingUtils.java`), iOS `UIViewContentMode` (`ScaleToFill`/`ScaleAspectFit`/`ScaleAspectFill`/`Center`, `LynxConverter (UIViewContentMode)` in `lynx/platform/darwin/ios/lynx/ui/image/LynxUIImage.mm:2063-2081`), and web-platform literally emits CSS `object-fit: fill/contain/cover` plus a `position:absolute` no-scale rule for `center` (`lynx-stack/packages/web-platform/web-elements/src/elements/XImage/x-image.css:38-53`). There is no Lynx equivalent of CSS `object-fit: scale-down` or `none` (as distinct from `center`) in any of the three stacks — a small, low-risk feature gap, not a divergence.

`cap-insets` (9-patch/9-slice stretching, `lynx/platform/android/.../image/NinePatchHelper.java`, `LynxNinePatchImageProcessor.m` on iOS) has no representation at all in web-platform's `x-image` — it is a native-only, non-W3C feature; the closest CSS primitive is `border-image-slice`/`border-image-width`/`border-image-repeat`, which only stretches border regions (not arbitrary interior tiling the way Lynx's cap-insets can), so a literal port isn't possible — lynx-vello should implement 9-slice compositing directly in the paint layer rather than trying to force it through a `border-image` shim.

Two real, per-platform default divergences worth flagging as risk (not clean W3C-vs-Lynx divergences, but native-Lynx internal inconsistency lynx-vello should normalize away by picking one behavior): (1) iOS's `LynxConverter (UIViewContentMode) toUIViewContentMode:` treats a *null* mode string as `aspectFill`/cover (`LynxUIImage.mm:2065-2080`), whereas the component's own `initProperties` default and Android's default are both `fill`/`FIT_XY`/`ScaleToFill` (`LynxUIImage.mm:284`, `LynxImageManager.java:134`) — i.e. `<image>` with no `mode` attribute renders as `fill` on both platforms in practice, but any other native code path that reuses that converter with a genuinely-nil value would silently diverge to `cover`. (2) `placeholder` reset semantics changed under a fix-flag (`enableImagePlaceholderResetFix`) so that clearing `placeholder` while `src` has already loaded does *not* blank the image (`LynxUIImage.mm:1314-1348`) — a subtlety worth matching exactly since it affects what a consumer sees mid-transition.

Lifecycle events are simply named `load` and `error` everywhere (Android: `LynxImageManager.EVENT_LOAD`/`EVENT_ERROR = "load"/"error"`, `lynx/platform/android/.../image/LynxImageManager.java:229-231`; iOS: `kLynxImageEventLoad`/`kLynxImageEventError`, `lynx/platform/darwin/ios/lynx/ui/image/LynxImageManager.mm:12-13`; web-platform: native DOM `load`/`error` events teleported 1:1 from the inner `<img>`, `lynx-stack/.../XImage/ImageEvents.ts:21-62`) — these are the ReactLynx `bindload`/`bindload`-equivalent hooks (JSX prop is `bindload`/`binderror`, mapped through the standard Lynx event-binding layer, not something image-specific). `load` detail payload is `{width, height}` (decoded/natural pixel dimensions) on all three platforms; `error` detail is `{errMsg, error_code, lynx_categorized_code}` on native (categorized via `ImageErrorCodeUtils.checkImageExceptionCategory`, buckets: user/design 1000s, network 1100s, decode/pic-source 1200s) vs. an empty `{}` detail on web-platform (browser doesn't expose a categorized decode-error code to JS) — lynx-vello should still emit `{}`-shaped detail by default for web-compat, and add categorized codes as an additive superset for native-parity consumers. Animated-image (GIF/WebP/APNG) loop-count and `startplay`/`currentloopcomplete`/`finalloopcomplete` custom events are native-only (Fresco `AnimatedDrawable2` callbacks, `LynxImageManager.java:439-461`); web-platform has no code for these at all (relies on native GIF looping in `<img>`, which has no equivalent lifecycle hooks) — a Lynx-only extension lynx-vello can only support via its own animated-codec + timer loop, not by any browser-standard mechanism.

Prefetch (`lynx.prefetchImage`/priority+cache-target API) exists only on Android (`LynxImageService.prefetchImage`, disk vs. bitmap cache target, `Priority.LOW/MEDIUM/HIGH`, `lynx/platform/android/lynx_service/lynx_service_image/.../LynxImageService.java:268-350`) with no equivalent found on iOS or web-platform in this pass — treat as Extended/Android-only, implement in lynx-vello's cache layer as a generic "warm the cache for this URL" op regardless of platform origin.

**W3C divergence flags:**
- `cap-insets` (9-slice) has no direct CSS equivalent (`border-image-slice` is the closest but semantically narrower — border-only, not arbitrary interior stretch). Not a "Lynx does it wrong" situation, just an extension with no standard mapping; implement natively rather than forcing a CSS shim.
- No genuine other Lynx-vs-W3C semantic divergence found in this subsystem: `object-fit` mapping, relative-URL resolution against the template's own base URL, and `load`/`error` event semantics are all already W3C/DOM-aligned in both native and web-platform Lynx. This subsystem is lower-risk for the "diverges from web platform" concern than most other Lynx built-ins.
