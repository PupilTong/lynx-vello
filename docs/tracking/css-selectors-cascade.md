# CSS selector engine, specificity, cascade & custom properties

> Research: multi-agent sweep over `lynx/` and `lynx-stack/` (see [AGENTS.md](../../AGENTS.md) for the reference-repo shorthand and the W3C-first standards policy). Supersedes the earlier stub.

### CSS Selector Engine: Combinators, Pseudo-Classes/Elements, Specificity & Cascade Matching

Lynx C++ actually contains **two parallel selector subsystems**, and the newer one only *parses* far more than it *matches*. This is the single most important finding for this section: lynx-vello must decide which behavior surface it is cloning, because the "spec" (grammar accepted) and the "ground truth" (runtime match result) diverge substantially in `lynx/core/renderer/css/ng/matcher/selector_matcher.cc`.

**Two engines, gated by `CSSFragment::enable_css_selector()`** (`lynx/core/renderer/css/css_fragment.h:76`, checked at `lynx/core/renderer/dom/style_resolver.cc:544,578,610,641`, `lynx/core/renderer/dom/fiber/fiber_element.cc:4903,5806`):
- **Legacy/"css_og" path** (`enable_css_selector() == false`, historically the ReactLynx2 default): compound-selector-only string matching with no combinators and no pseudo-classes, implemented in `lynx/core/renderer/dom/attribute_holder.cc` (`ContainsSelector`, `ContainsIdSelector`, `ContainsClassSelector`, `ContainsTagSelector`, `ContainsAttributeSelector`) plus `lynx/core/renderer/dom/selector/matching/attribute_selector_matching.{h,cc}` for the `[attr]`/`[attr=v]`/`[attr*=v]`/`[attr^=v]`/`[attr$=v]` grammar. lynx-stack's web target mirrors this exact flag as `config_enable_css_selector` (`lynx-stack/packages/web-platform/web-core/ts/client/decodeWorker/cssLoader.ts:20`, `lynx-stack/packages/web-platform/web-core/src/template/template_sections/style_info/decoded_style_data.rs:48,77`) — when false, single-class rules are pulled out into a `css_og_css_id_to_class_selector_name_to_declarations_map` side table instead of real CSS rules (`lynx-stack/.../style_info_decoder.rs:77-108`).
- **NG path** (`enable_css_selector() == true`, the current default in lynx-stack tests, e.g. `lynx-stack/packages/web-platform/web-core/tests/server-compat.spec.ts:23`): full combinator + pseudo grammar in `lynx/core/renderer/css/ng/selector/{lynx_css_selector.h,css_selector_parser.cc,lynx_css_selector_list.cc}` and matched at runtime by `lynx/core/renderer/css/ng/matcher/selector_matcher.cc`.

**Critical gap in the NG matcher**: `SelectorMatcher::MatchSimple` (`selector_matcher.cc:113-135`) only has cases for `kTag`, `kClass`, `kId`, `kPseudoClass`, `kPseudoElement` — every `kAttribute*` match type (`kAttributeExact/Set/Hyphen/List/Contain/Begin/End`) falls through to `default: return false`. Attribute selectors (`[attr=val]` etc.) are fully parsed (`css_selector_parser.cc:417-454,584-619`, including the Level-4 `i`/`s` case-sensitivity flags) but **can never match** through the NG runtime path. Likewise `SelectorMatcher::MatchPseudoClass` (`selector_matcher.cc:152-173`) only implements `kPseudoNot`, `kPseudoHover`, `kPseudoActive`, `kPseudoFocus`, `kPseudoRoot` — `:nth-child`, `:nth-of-type`, `:nth-last-child`, `:nth-last-of-type`, `:first-child`, `:last-child`, `:only-child`, `:first-of-type`, `:last-of-type`, `:only-of-type`, `:is()`, `:where()`, `:has()`, `:lang()`, `:dir()`, `:empty`, `:checked`, `:enabled`, `:disabled`, `:default`, `:link`, `:visited` all fall through to `return false` even though they parse successfully and get correct specificity values. `LynxCSSSelector::MatchNth`/`LynxCSSSelectorExtraData::MatchNth` (`lynx_css_selector.cc:172-174`, `lynx_css_selector_extra_data.cc:24-40`) implement the standard `an+b` arithmetic correctly but are **dead code** — never called from the matcher. Confirmed via `lynx/core/renderer/css/ng/matcher/selector_matcher_test.cc` (only `:root`, `:focus`, `:active`, `:hover`(as non-rightmost only), `:not()` are exercised) and `lynx/core/renderer/css/ng/selector/css_selector_parser_test.cc` (no `:nth-child`/`:is`/`:where` test cases at all).

**Combinators** (`kDescendant`/`kChild`/`kDirectAdjacent`/`kIndirectAdjacent`/`kSubSelector`/`kUAShadow`, `lynx_css_selector.h:67-78`) are fully and correctly implemented in `MatchForRelation` (`selector_matcher.cc:60-111`), matching W3C right-to-left compound-selector walking semantics (descendant walks all ancestors, child checks immediate parent, `+`/`~` walk previous siblings). `kUAShadow` is an implicit combinator Lynx synthesizes for `::placeholder`/`::selection` (`css_selector_parser.cc:36-51,746-793`) to cross from the real element to a synthetic pseudo-element node (`StyleNode::PseudoElementOwner()`).

**`::before`/`::after` generated content does not exist in native Lynx at all.** `PseudoIdBefore`/`kPseudoIdAfter`/`kPseudoBefore`/`kPseudoAfter` are declared enum values (`lynx_css_selector.h:23-31,80-118`) referenced only inside the parser's pseudo-name table (`css_selector_parser.cc:171,489`) — there is no `content` CSS property anywhere in Lynx's property table, no generated-content box creation, and no other reference to these pseudo IDs in the whole `core/renderer` tree outside the parser. By contrast, **lynx-stack's web target passes selectors and declarations through almost verbatim into a real `<style>` sheet** (`lynx-stack/packages/web-platform/web-core/src/template/template_sections/style_info/style_info_decoder.rs`), and `content` IS a recognized property there (`css_property.rs:130`) — meaning `::before`/`::after {content: ...}` would actually render via the browser's native engine on Lynx-for-Web even though it is inert on native Lynx. This is a real behavioral divergence between the two existing Lynx runtimes, not just a spec gap.

**Scoped CSS ("css_scope") is not a selector-rewriting/shadow-DOM mechanism — it's an attribute selector.** The `swc_plugin_css_scope` Rust crate (`lynx-stack/packages/react/transform/crates/swc_plugin_css_scope/lib.rs`) only tags each CSS import with a numeric `cssId` (via `?cssId=N` query + a `@jsxCSSId` leading comment) at build time; it does zero selector text mutation. The actual isolation enforcement happens later, at web-bundle-decode time, by **appending a real W3C `:where([l-css-id="N"])` pseudo-class wrapping an attribute selector** to every compound selector (`lynx-stack/packages/web-platform/web-core/src/template/template_sections/style_info/style_info_decoder.rs:193-221`), plus a parallel `:not([l-e-name])` / `[l-e-name="entry"]` attribute selector for multi-entry-point isolation (`style_info_decoder.rs:170-191`). `:where()` is deliberately used (over `:is()`) because it contributes **zero specificity**, so scoping never perturbs the author's intended cascade ranking — this is correct, standard CSS-Scoping-flavored technique and fully W3C-compliant since it's real browser CSS. lynx-vello's native matcher must implement `:where()` support (with 0 specificity, matching `CalcSpecificityForSimple`'s `kPseudoWhere: return 0` at `lynx_css_selector_impl.cc:103-104`) if it wants to reuse this exact scoping strategy natively, since Lynx's own NG matcher does not currently execute `:where()` matching (see gap above) even though it computes its specificity correctly.

**Cascade/specificity ranking is W3C-compliant.** `LynxCSSSelector::CalcSpecificity`/`CalcSpecificityForSimple` (`lynx_css_selector_impl.cc:72-135`) use the standard three-tier encoding (id=`0x010000`, class/attribute/pseudo-class=`0x000100`, type=`0x000001`), with `:not()`/`:is()`/`:has()` contributing their argument list's max specificity and `:where()` contributing 0 — all per CSS Selectors Level 4 §specificity. Final rule ordering is `std::sort`-equivalent (`base::InsertionSort`) by `(specificity, position)` ascending in `lynx/core/renderer/dom/style_resolver.cc:803-810,875`, where `position` encodes `(rule_set_level << bits) + source_order` so ties break by source order — standard "cascade origin/order, then specificity, then source order" (simplified since Lynx has no CSS layers/`@layer`). `!important` is handled as a wholly separate bucket (`matched_important_style_map`) merged after the normal bucket (`style_resolver.cc:551-569,657-660`) — correct override semantics.

**Dynamic pseudo-classes are driven by explicit platform-side state pushes, not continuous DOM introspection.** `:hover`/`:active`/`:focus`/`:placeholder`/`:selection` are bitflags (`kPseudoStateHover=1`, `kPseudoStateActive=1<<3`, `kPseudoStateFocus=1<<6`, plus `*Transition` variants for animated state, `lynx/core/renderer/css/css_property.h:26-36`) stored on `AttributeHolder` (`lynx/core/renderer/dom/attribute_holder.h:385-412`) and toggled by `TouchEventHandler::HandlePseudoStatusChanged` (`lynx/core/renderer/events/touch_event_handler.cc:423-434`), which is invoked by the native shell/platform layer (not by the selector engine watching real pointer state) and triggers `Element::OnPseudoStatusChanged` → `RuleInvalidationSet`-driven restyle (`lynx/core/renderer/dom/fiber/fiber_element.cc:4887-4947`). Because Lynx targets touch devices, `:hover` in practice is commonly synthesized from touch-start rather than true pointer-hover — lynx-vello should verify per-platform what gesture maps to `:hover` vs `:active` rather than assuming desktop mouse semantics.

**`:not()` list semantics are W3C Level-4-correct ("forgiving" per-selector, non-forgiving overall isn't applicable since Lynx's parser rejects the whole rule on any unparseable selector at parse time, not per test).** `MatchPseudoNot` (`selector_matcher.cc:137-150`) iterates the selector list and returns true only if none match — confirmed by test data `:not(:active, :hover)` → true when neither state is set, `:not(text)` → true for non-`text` tag (`selector_matcher_test.cc:77-80`).

### CSS Custom Properties (`--foo: ...`) and `var()` Resolution

This is architecturally a **separate subsystem** from ordinary property inheritance, with its own always-on inheritance rule, its own dedicated data model (`CustomPropertiesMap` / `CSSValue::VARIABLE`), and — critically — **two parallel, functionally-overlapping runtime implementations** inside the native engine: a legacy Mustache-style `{{--name}}` string-substitution path, and a newer, spec-shaped `var(--name, fallback)` path gated behind a still-off-by-default flag. The web target (`lynx-stack/web-core`) implements neither: it defers entirely to the browser's native CSS custom-property engine.

**Architecture summary (native engine, `lynx/core/renderer/css/`):**
- `CSSValue` carries an `is_variable_` bit and (lazily) a `base::Vector<VarReference>` describing the byte ranges of each `{{...}}` or `var(...)` occurrence in its raw string, plus an optional per-reference `fallback` string (`lynx/core/renderer/css/css_value.h:64-77`, `:206-333`).
- Two independent syntaxes are parsed into the *same* `VarReference` struct: `{{--name}}` (legacy, `offset=2`, built by `CSSValue::ToVarReference()`, `lynx/core/renderer/css/css_value.cc:754-813`) and `var(--name, fallback)` (spec syntax, `offset=4`, built by `CSSStringParser::ParseVariable`/`ParseVarReference`, `lynx/core/renderer/css/parser/css_string_parser.cc:3109-3174`). The web build-time compiler (`lynx-stack/packages/tools/css-serializer/src/parse.ts:20-118`) rewrites authored `var(--name)` into the legacy `{{--name}}` form for the *non-web* (native mobile) `.lynx.bundle` target and explicitly **rejects nested `var()`** as a hard compile error (`nested var() is not allowed in Lynx`); the `.web.bundle` target (consumed by this repo) instead keeps `var(...)` as literal CSS text (`lynx-stack/packages/web-platform/web-core/src/template/template_sections/style_info/style_info_decoder.rs:985-1018`).
- `CSSVariableHandler` (`lynx/core/renderer/css/css_variable_handler.{h,cc}`, ~213 lines total inc. license header) is the entry point used from `StyleResolver::HandleCSSVariables` (`lynx/core/renderer/dom/style_resolver.cc:737-770`). It has two code paths selected by `enable_fiber_arch_` (constructor arg, default `false`; instantiated with `true` from `style_resolver.cc:744,1809,2008`):
  - **Legacy path** (`HandleCSSVariables`, `css_variable_handler.cc:19-77`): for each `{{...}}`-bearing `CSSValue` without a resolved `VarReference` (`NeedsVariableResolution()==false`), calls `GetCSSVariableByRule` which walks the format string character-by-character, replacing each `{{key}}` via `AttributeHolder::GetCSSVariableValue` (DOM-parent walk, see below), falling back first to a per-declaration `default_value_map` (parsed from CSS at build time, mapping var-name→fallback text) then to the single `default_props`/`GetDefaultValue()` string. **If the resolved variable value is empty, the whole declaration falls back to `default_props`** (not per-var fallback) — i.e., legacy fallback is coarse-grained (`css_variable_handler.cc:183-186`, confirmed by unit test `FormatStringWithRule7/8` in `css_variable_handler_unittest.cc:129-162`).
  - **Fiber/new path** (`ResolveCSSVariables`→`CSSValue::SubstitutionResolved`, only active when `enable_fiber_arch_==true`, itself gated by the separate config flag `enable_css_inline_variables_`, default `false`, `lynx/core/renderer/css/dynamic_css_configs.h:40`, checked via `Element::IsCSSInlineVariablesEnabled()` at `lynx/core/renderer/dom/element.cc:1914-1917`): resolves each `var(--name, fallback)` reference against a `CustomPropertiesMap` that was pre-substituted (all-vars-resolved-to-strings) by `CollectCustomProperties`/`CSSValue::SubstituteAll` (`lynx/core/renderer/dom/fiber/fiber_element.cc:5746-5798`, `lynx/core/renderer/css/css_value.cc:535-547`). Per-reference fallback IS honored here when the variable is simply undefined (`ResolveFallbackResolved`, `css_value.cc:732-743`).
- **Cycle detection**: `CSSValue::CycleDetector` runs Tarjan's SCC algorithm over the *entire* custom-property map before each substitution pass (`css_value.cc:407-505`). Any variable participating in a cycle (including a pure self-reference, `var(--self)` referencing itself) resolves to the **empty string**, and — per `css_value_unittest.cc:409-427` (`ComplexCycleWithFallback`) and `:465-503` (`CycleWithFallbackCorrectBehavior`) — **the per-`var()` fallback value is deliberately NOT used for cyclic variables**, only for genuinely-undefined ones. `max_depth` (default 10) is a separate, non-cycle recursion-depth guard that also yields empty string when exceeded (`css_value_unittest.cc:230-286`, `DepthLimit`).
- **Inheritance is unconditional and separate from `enableCSSInheritance`.** `AttributeHolder::GetCSSVariableValue` (legacy path) walks `HolderParent()` all the way to the root regardless of any flag (`lynx/core/renderer/dom/attribute_holder.cc:191-208`). For the fiber/new path, `FiberElement::CollectCustomProperties` copy-on-write-shares the parent's resolved `CustomPropertiesMap` down the tree unconditionally (`fiber_element.cc:5746-5798`), and at the `ComputedCSSStyle` layer `StyleResolver::InheritParentStyle` calls `computed_style.InheritCustomPropertiesFrom(*parent_style)` **before** checking `element()->IsCSSInheritanceEnabled()`, whereas the very next lines gate `InheritNormalPropertiesFrom`/`InheritResolvedValuesFrom` behind that same flag and `return` early if it's off (`lynx/core/renderer/dom/style_resolver.cc:1688-1714`). This is the definitive evidence: **custom-property inheritance is architecturally independent of, and always-on regardless of, `enableCSSInheritance`** (which defaults to `false` and only gates the small hardcoded allowlist of standard inheritable properties like `color`/`font-*`/`direction`/`cursor`, see `ComputedCSSStyle::InheritNormalPropertiesFrom`, `computed_css_style.cc:4986-5128`).
- Declaration/registration: any declaration whose name starts with `--` and has length > 2 is recognized as a custom property (`CSSProperty::IsCustomProperty`, `lynx/core/renderer/css/css_property.h:299-301`) — this matches the W3C custom-property naming rule exactly. There is **no `@property`/typed-registration equivalent** anywhere in the native engine (no hits for `@property`/`registerProperty` under `core/renderer`); all custom properties are late-bound, untyped strings, i.e. equivalent to CSS Custom Properties with `syntax: "*"` and `inherits: true` always, `initial-value` undefined.
- JS/dynamic updates: `AttributeHolder::UpdateCSSVariableFromSetProperty` / `UpdateCSSInlineVariables` provide a "from JS" override map (`css_variables_from_js_`) that is checked *before* the CSS-cascade-derived map in both `GetCSSVariableValue` (legacy) and inline-variables resolution — i.e., inline/JS-set custom properties win over stylesheet-derived ones, mirroring normal CSS cascade specificity for `style=""` vs stylesheet (`attribute_holder.cc:138-208`, `attribute_holder.h:279-320`). No dedicated public JS API (e.g. `element.setProperty('--x', v)`) was found wired into `packages/react/runtime/src/lynx-api.ts`; custom properties are exposed as ordinary inline-style keys.
- Keyframe/animation interaction: `@keyframes` blocks can themselves declare custom properties per-frame (`CSSKeyframesCustomPropertyContent`, populated in `CSSStyleUtils::UpdateCSSKeyframes`, `lynx/core/renderer/css/css_style_utils.cc:1431-1450`), and any `var()` reference inside a keyframe value is resolved against the **union of the keyframe's own custom properties (highest priority) and the element's inherited custom properties** via `CSSStyleUtils::ResolveCSSKeyframeValueWithCustomProperties` (`css_style_utils.cc:1393-1429`) — this is a bespoke resolution order not found in the non-animation cascade path.
- `ComputedCSSStyle` stores both a "raw" (`raw_custom_properties_`, may still contain unresolved `var()` refs) and a "resolved" (`resolved_custom_properties_`, all-strings-substituted, produced by `FinalizeCustomProperties()` → `CSSValue::SubstituteAll`) copy of the custom-property map, both copy-on-write refs (`starlight::DataRef`), confirmed by `computed_css_style.cc:4980-5192` and exercised in `computed_css_style_unittest.cc:527-622` (`FinalizeCustomPropertiesResolvesVariables`, `CopyFromPreservesComputedCssStateIndependently`, `InheritHelpersCopyOnlyRequestedState`).
- Whitespace/`!important` edge cases observed in tests: a fallback value is appended **including the leading space** after the comma in `var(--x, fallback)` (`var(--primary, blue)` with `--primary` undefined resolves to `" blue"`, not `"blue"` — `css_value_unittest.cc:134-144`), and any `!important` suffix present in the *resolved* string is stripped post-substitution (`StripImportantFromString`, `css_value.cc:559-569,645,714`) — i.e. `!important` cannot leak in through a substituted custom-property value.

**Web target (`lynx-stack/web-core`, what this repo's decoder output actually feeds):** the Rust/wasm `style_info_decoder.rs` re-emits declaration values **as literal CSS text** into a generated stylesheet string injected via `<style>` (confirmed by `style_content` being a plain `String` built by directly appending selector/declaration text, `style_info_decoder.rs` throughout, and by round-trip test `test_css_var_in_property` at lines 972-1020 asserting the decoded output literally contains `"background-color:var(--main-bg-color, red);"` and `"border:1px solid var(--border-color);"` — i.e. `var()` and fallback text pass through **completely unresolved and unmodified**). Web-core additionally uses its own internal `--lynx-*` custom properties as an implementation trick for shorthand-like Lynx properties (`display:linear`, `linear-direction`, `linear-weight`, text background clipping, `rpx` units via `calc(100 * var(--rpx-unit))`) — see `lynx-stack/packages/web-platform/web-core/src/style_transformer/inline_style.rs:88-303` test cases. This means: **for the web target, all `var()`/custom-property registration, inheritance, fallback, and cycle-handling semantics are delegated entirely to the real browser CSSOM** — web-core does zero custom substitution logic itself. A noted quirk in the decoder's own test comments: an optimization path can cause a custom property's *name* to be lost/mangled for "unknown properties" in some encodings (`style_info_decoder.rs:1012-1015`, commented-out assertion), worth re-verifying against real fixtures before relying on `--name` round-tripping losslessly through the wire format.

**Implication for lynx-vello:** since lynx-vello uses **stylo** (a real W3C-standards CSS engine, the same one Firefox uses) rather than web-core's browser pass-through trick, custom properties should be implemented using stylo's native, fully-spec-compliant custom-property/`var()` support (unconditional inheritance, per-reference fallback, proper cycle → "invalid at computed-value time" handling per the CSS Custom Properties spec) rather than porting either of Lynx's native bespoke mechanisms. The one behavior lynx-vello must deliberately *replicate* (not "fix") is that **custom-property inheritance in Lynx is always on**, even when `enableCSSInheritance` is off for normal properties — this happens to already match the W3C standard (custom properties are `inherits: true` by definition), so no divergence flag is needed there. The legacy `{{}}`-template mechanism and its coarse-grained "whole declaration falls back to `default_props`" semantics is Lynx-bundle/native-target-only formatting baggage that this repo's web-bundle decoder does not need to reproduce, since `.web.bundle` payloads keep standard `var()` syntax.
