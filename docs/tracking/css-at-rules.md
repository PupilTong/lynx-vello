# CSS at-rules: @media, @font-face, @supports

> Research: multi-agent sweep over `lynx/` and `lynx-stack/` (see [AGENTS.md](../../AGENTS.md) for the reference-repo shorthand and the W3C-first standards policy). Supersedes the earlier stub.

### Media Queries & Viewport/Device Media-Feature Evaluation

**Architectural summary.** Lynx (C++ engine, `develop` branch) has a fully-built, standards-modeled `@media` subsystem under `core/renderer/css/ng/media_query/` plus a recursive-descent parser at `core/renderer/css/ng/parser/media_query_parser.{h,cc}`. It was added very recently (commits `aed796c75` "[Feature] Add CSS media query evaluator" and `49483eac5` "[Feature][CSS] Add UpdateColorScheme API for dynamic prefers-color-scheme support", both dated 2026-06-08 on `develop`) — this is new, not legacy, code, fully unit-tested (`media_query_test.cc`, `media_query_parser_test.cc`, `media_query_evaluator_test.cc`). Critically, **this subsystem is wired only into the *native* `.lynx.bundle` / `ConditionRule` pipeline** (`core/renderer/css/ng/style/condition_rule.{h,cc}`, `rule_set.cc`'s `MatchStyles`, `core/renderer/dom/style_resolver.cc`'s `BuildMediaQueryEvaluator`/`GetCSSStyleNew`). It is **not reachable from the web-bundle format at all**: lynx-stack's web-core rkyv `RawStyleInfo` schema (`lynx-stack/packages/web-platform/web-core/src/template/template_sections/style_info/raw_style_info.rs`) defines `RuleType` with exactly three variants — `Declaration = 1`, `FontFace = 2`, `KeyFrames = 3` — with no media/condition-rule variant, and the CSS encoder (`lynx-stack/packages/web-platform/web-core/ts/encode/encodeCSS.ts`) and tokenizer (`lynx-stack/packages/web-platform/web-core/src/css_tokenizer/`) contain zero references to `media`/`@media`. lynx-vello's own decoder (`crates/lynx-template-decoder/src/style_info.rs`, `RuleType` enum, doc comment "Mirrors web-core's `RuleType`") faithfully mirrors this same 3-variant schema. **This means: as currently specified, the `.web.bundle` format this project decodes has no wire representation for `@media` at-rules; any `@media` block in ReactLynx source CSS is presumably dropped/unsupported when compiled to the web target today.** This is the single most important finding for this section — treat it as a confirmed gap in the artifact format lynx-vello consumes, not a coverage gap in our reading.

Given that, lynx-vello has two paths: (a) match today's web-bundle behavior (i.e., `@media` rules never apply / are stripped, since ReactLynx-for-web apps can't rely on them), or (b) since lynx-vello is a *native* rewrite that could parse CSS text directly rather than only consuming the existing web-bundle wire format, adopt the C++ engine's model (which is well-designed and standards-aligned) as the target behavior for ReactLynx compatibility once media queries are actually authored. The C++ evaluator model below is the correct **behavioral reference** for what "matching" should mean; the table also flags where the *native* engine's own behavior is non-standard, incomplete, or effectively unobservable in the shipped app (which functionally matches today's web-bundle gap even for the native engine, e.g. hover/pointer always `unknown`).

Key architecture notes (prose, not enumerable as rows):
- **AST shape**: `MediaQuerySet` (comma-separated list) → `MediaQuery` (`only|not` restrictor + media-type string + optional condition) → `MediaQueryExpNode` tree with node kinds `kFeature | kNested | kNot | kAnd | kOr` (`lynx/core/renderer/css/ng/media_query/media_query_exp.h`). Grammar follows CSS Media Queries Level 4 including the `A op1 name op2 B` range syntax (`lynx/core/renderer/css/ng/parser/media_query_parser.h:8-21` grammar comment, `lynx/core/renderer/css/ng/media_query/media_feature.h:235-263` `MediaFeature` range fields). Values, node types, and operators are serialized with explicit stable numeric tags for the native template binary (`media_query_exp.h:26-34`, `media_feature.h:32-129`) — i.e. the native `.lynx.bundle` format *does* have room for this, unlike the web format.
- **Evaluation is a pure, cheap, on-demand pass**: `MediaQueryEvaluator` just holds a `MediaValues` snapshot and is rebuilt fresh on every `GetCSSStyleNew` call when the stylesheet's `RuleSet` flags say `kHasMediaQuery` (`lynx/core/renderer/dom/style_resolver.cc:814-836, 879-892`). There is **no persistent `MediaQueryList`/listener object and no JS-exposed `matchMedia()` API** — grepped both repos, zero hits outside of the browser DevTools helper page (`lynx-stack/packages/web-platform/web-explorer/index.ts:103`, which just reads the *browser's* `window.matchMedia`, unrelated to Lynx's own engine). ReactLynx apps cannot programmatically query media state; `@media` is purely a CSS-conditional-block mechanism.
- **Re-evaluation triggers**: `DynamicCSSStylesManager` defines update-flag bits `kUpdateViewport`, `kUpdateScreenMetrics`, `kUpdateRem`, `kUpdateEm`, `kUpdateColorScheme`, `kUpdateFontScale` (`lynx/core/renderer/css/dynamic_css_styles_manager.h:61-89`). `FiberElement::UpdateDynamicElementStyleForNewPipeline` (`lynx/core/renderer/dom/fiber/fiber_element.cc:5161-5246`) computes a `kMediaQueryEnvMask` (viewport | screen-metrics | rem | em | color-scheme) and — **only at component-fragment boundaries** (`is_component()` check) whose stylesheet `HasMediaQueryRules()` — forces a full re-resolve of that subtree. `ElementManager::UpdateColorScheme(int)` (`lynx/core/renderer/dom/element_manager.cc:697-713`) is the only host-driven entry point for `prefers-color-scheme`: it validates the int is 0/1, updates `LynxEnvConfig`, triggers `kUpdateColorScheme`, and calls `RequestResolve`. This is plumbed all the way through platform embedders (Android `LynxView`/`LynxViewBuilder`/`LynxColorScheme.java`, iOS `LynxView.mm`+`LynxViewEnum.h`, per commit `49483eac5`'s diffstat) — i.e., the **host app** (native shell), not the OS `prefers-color-scheme` media feature automatically, decides and pushes the color scheme value into Lynx. There is no OS dark-mode auto-detection wired in the C++ core itself (that would live in each platform's embedder layer, outside scope of what was read here — not confirmed either way for Android/iOS auto-detection).
- **`MediaValues` snapshot construction** (`lynx/core/renderer/dom/style_resolver.cc:814-836`, `BuildMediaQueryEvaluator`): viewport width/height come from `LynxEnvConfig::ViewportWidth()/Height()` if definite, else falls back to `ScreenWidth()/ScreenHeight()`; device-pixel-ratio from `LynxEnvConfig::DevicePixelRatio()`; root font-size from the tree root element's computed font size; `font_size` (for `em`) from the *owning element's* computed font size (so `em`-unit media features are contextual per node, matching CSS spec intent for container-relative rem/em in media features). **`hover` and `pointer` are never set anywhere in production code** — grepped all of `core/renderer/` for `SetHover`/`SetPointer`/`MediaTristate::kPresent` outside the media_query directory itself and found zero call sites; they default to `MediaTristate::kUnknown` forever, meaning in the shipped renderer `(hover)`, `(hover: hover)`, `(hover: none)`, `(pointer: coarse|fine|none)` **always evaluate false** (per `EvalHoverFeature`/`EvalPointerFeature`, `lynx/core/renderer/css/ng/media_query/media_query_evaluator.cc:367-390`, which only pass when `MediaTristate::kPresent`/`kNone` is explicitly set — a state nothing produces). This is a currently-dead/aspirational feature in the engine.
- **Empty/invalid list semantics** (spec-faithful): empty or whitespace-only media-query text parses to an empty `MediaQuerySet` and matches everything (`MediaQueryEvaluator::Eval(const MediaQuerySet&)`, `lynx/core/renderer/css/ng/media_query/media_query_evaluator.cc:164-172`; test `EmptySetMatches`/`EmptyTextParsesAndMatches`). Wholly-unparseable text collapses to a single `not all` entry (never matches) rather than an empty set (`MediaQueryParser`, tests `WhollyInvalidInputProducesNotAllPlaceholder`, `InvalidTextDoesNotMatch`) — this matches the CSSOM spec's "unknown media query behaves as not all" rule.
- **Media type support is minimal but standard-shaped**: only `screen` (always matches — "the renderer is always a screen-like surface", `media_query_evaluator.cc:181-183`) and `all`/empty type match; every other type name (`print`, `speech`, custom idents) always fails, so `not print` always matches and `not screen` never matches on this renderer (tests `PrintDoesNotMatch`, `NotPrintMatches`, `NotScreenDoesNotMatch`). This is W3C-compliant behavior for a screen-only UA (Media Queries §2.3 media-type table explicitly allows a UA to treat itself as screen-only).

### @font-face Rule Parsing & Web Font Loading/Registration

This is a genuinely separate subsystem from font-* text-run properties: it spans (1) a CSS-grammar `@font-face` descriptor parser, (2) a build-time encoder that serializes parsed font-face rules into the template binary, (3) a runtime decoder/registration pipeline that is **lossy** relative to the parser, and (4) a JS-visible `lynx.addFont()` native API that is a *separate, simpler* registration path bypassing CSS parsing entirely. Native Lynx and Lynx-for-Web (web-core) diverge sharply here: native Lynx has a full modern `@font-face` descriptor grammar (weight/stretch/style ranges, unicode-range, variation-settings) that is nevertheless discarded before reaching platform font managers, while web-core has no custom `@font-face` parsing at all — it re-serializes verbatim CSS text and lets the browser's native font engine do everything, and has **no** `addFont` implementation whatsoever.

**Architecture note (not a table-friendly item):** Lynx's `@font-face` pipeline has three representations that get progressively simplified:
1. **Parse-time (encoder, build tool)**: `lynx/core/renderer/css/ng/parser/font_face_parser.cc` (`css::FontFaceParser::Parse`) parses the full W3C CSS Fonts Level 4 `@font-face` descriptor grammar from a `name→value` descriptor map into a rich `css::FontFaceRule` (`lynx/core/renderer/css/ng/font_face/font_face_rule.h`): family, ordered `src` list (each entry `{is_local, uri}` plus ignored `format()`/`tech()` hints), `font-weight` range (keyword `normal`/`bold` or numeric 1-1000, min≤max normalized), `font-stretch` range (9 CSS keywords or percentage), `font-style` (`normal`/`italic`/`oblique <angle> <angle>` with correct default 14deg and unit conversion deg/grad/rad/turn), `font-variation-settings` (4-char axis tags + float values), and `unicode-range` (parsed by the tokenizer's dedicated `kUnicodeRangeToken`, supporting `U+XXXX-YYYY` and wildcard `U+X??` forms). This rule round-trips through `ToLepus()`/`FromLepus()` (weight/stretch packed as `(min<<16)|max` u32) for storage in the template binary (`lynx/core/renderer/css/ng/font_face/font_face_rule.cc`).
2. **Encode-time**: `lynx/core/template_bundle/template_codec/binary_encoder/css_encoder/css_rule_parser.cc` (`CSSRuleParser::ParseFontFaceRule`) is invoked from the top-level stylesheet dispatch (`ParseCSSRules`), from `@media`/`@supports` child-rule dispatch (`ParseConditionRule`), and from `@layer` child-rule dispatch (`ParseLayerRule`) — i.e. `@font-face` is a legal nested rule everywhere in native Lynx's grammar. It builds an `encoder::LynxStyleRuleFontFace{family, font_face_rule: lepus::Value}` and is encoded via `EncodeCSSFontFaceRule` (`template_binary_writer_impl.cc:520`) as a length-prefixed `CSSRuleType::kFontFace` binary rule (forward-compatible skip-if-unknown framing). A **parallel legacy path** also exists: `GetFontFaceTokenMapForEncode()`/`EncodeCSSFontFaceTokenList`/`EncodeCSSFontFaceToken` encode `CSSFontFaceToken` (flat string attr map, from `lynx/core/template_bundle/template_codec/binary_encoder/css_encoder/css_font_face_token.{h,cc}`) — an older, string-map-only representation still emitted alongside the new one.
3. **Decode/runtime-time (the lossy step)**: `lynx/core/template_bundle/template_codec/binary_decoder/lynx_binary_base_css_reader.cc` decodes the rich `css::FontFaceRule` via `FontFaceRule::FromLepus`, then **immediately downgrades it** via `CreateLegacyFontFaceRule()` (line 45) to the flat runtime type `CSSFontFaceRule = std::pair<std::string /*family*/, std::unordered_map<std::string,std::string> /*attrs*/>` (defined in `lynx/core/public/layout_ctx_platform_impl.h:29`), populating only `font-family` and a re-serialized `src` string (`FormatFontSources`, `url(...)`/`local(...)` joined with `, `). **Weight/stretch/style ranges, unicode-range, and variation-settings are silently dropped at this step** — `layout_ctx_platform_impl.h:31-36` has a standing `TODO: using css::FontFaceRule` acknowledging this is known debt, not yet wired to platform layout contexts. Additionally, `@font-face` nested inside `@media`/`@supports` is decoded (`DecodeConditionRuleData`, line 367) but the parsed `family`/`token_list` are **thrown away with no use** — conditional (media-query-scoped) `@font-face` rules never reach the runtime font-face map at all. `@layer`-scoped font-face rules are similarly decoded-then-unused in the layer path (grep shows `DecodeCSSLayerRule`'s font-face decode call at line 486 has the same discard pattern).

**Runtime registration & lazy resolution**: Once decoded into `SharedCSSFragment::fontfaces_` (`CSSFontFaceRuleMap`, keyed by family), font-faces are **not** eagerly pushed to the platform. In the Fiber architecture, `TextElement::ResolveAndFlushFontFaces` (`lynx/core/renderer/dom/fiber/text_element.cc:232`) is triggered only when a `font-family` CSS property is actually resolved on a text node; it then calls `fragment->ForEachUnresolvedFontFaceMap(...)` and pushes **the entire unresolved font-face map** (not filtered by the specific family requested — an explicit `FIXME(linxs)` at line 239 acknowledges this imprecision) to `ElementManager::SetFontFaces()` → `LayoutCtxPlatformImpl::SetFontFaces(const CSSFontFaceRuleMap&)`, the abstract per-platform registration entrypoint (implemented by Android/iOS/Harmony `layout_context_*`). `ComponentElement::PrepareForFontFaceIfNeeded` (`component_element.cc:142`) does the same flush per-component-stylesheet in the Radon-Fiber compatibility path, with an inline comment explaining the historical Radon-vs-Fiber behavior difference (Radon flushed globally per-page during node parsing; Fiber flushes lazily per-text-node). Each fragment/stylesheet is marked resolved once via `MarkFontFacesResolved(bool)` to avoid re-flushing.

**Platform font loading (concretely observed only for Harmony, no in-tree Android/iOS font-loading source)**: `lynx/platform/harmony/lynx_harmony/src/main/cpp/font/font_face_manager.{h,cc}` re-parses the flattened `src` string itself (ad-hoc `url(`/`local(` bracket splitting in `FontFace::ParseAndAddSrc`, not reusing the already-parsed `FontSource` structs from step 1 — confirming the downgrade is real and irreversible by the time platform code runs). `LoadFontWithUrl` handles: `local()` sources are **unimplemented** (`TODO(linxs): support it later` — returns error state immediately); `url()` sources support data-URIs (decoded inline, synchronously) and HTTP(S) URLs (async via `LynxResourceRequest{..., kFont}` through the generic resource loader, with per-family loading-state dedup, a `FontFaceCache`, and pending-shadow-node queuing so nodes awaiting the font get marked dirty and re-laid-out once it arrives).

**`lynx.addFont()` native API** (a distinct, non-CSS registration path): JS call chain is `lynx.addFont(fontFace, callback)` (binding: `lynx/core/runtime/js/bindings/lynx.cc:298`, requires exactly 2 args, arg0 object, arg1 function) → `App::AddFont` (`js_app.cc:3159`) → `TemplateDelegate::AddFont` → `BTSRuntimeMediator::AddFont` (background-thread → engine-thread hop via `engine_actor_->ActAsync`, `bts_runtime_mediator.cc:162`) → `LynxEngine::AddFont` → `TemplateAssembler::AddFont` (`template_assembler.cc:1290`) → `ElementManager::AddFontFace(lepus::Value)` (`element_manager.cc:956`) → `MakeCSSFontFaceToken()` (`css_font_face_token.cc:26`, trims whitespace/quotes off keys/values, requires a `font-family` key) → wraps as a single-entry `CSSFontFaceRuleMap` → `delegate_->SetFontFaces(map)` (same platform entrypoint as the CSS path). The JS-visible shape is deliberately minimal: TS type `FontFace = {'font-family': string, 'src': string}` (`lynx/js_libraries/types/types/background-thread/lynx.d.ts:78`), and `lynx-core`'s wrapper (`lynx/js_libraries/lynx-core/src/lynx/lynx.ts:188`) validates both fields are strings and the callback is a function, then calls `getNativeLynx().addFont(font, callback)`. **No weight/style/stretch/unicode-range support exists in this path** — it can only ever produce a single unconditional font-face (one `src`, no variant matching). Also: **the callback is passed all the way down but nothing in the C++ chain we found ever invokes it** — registration is fire-and-forget with no observed success/failure signal back to JS.

**Lynx-for-Web (web-core) — architecturally simpler, and missing `addFont` entirely**: On the web target, `@font-face` is captured by the build-time `css-serializer` (`lynx-stack/packages/tools/css-serializer/src/parse.ts:307`, top-level `csstree` walk matching `Atrule` named `font-face`) as an **opaque list of `Declaration`s** (no descriptor-specific parsing — no weight/stretch/style/unicode-range interpretation at all; whatever the author wrote is preserved as CSS text tokens). Critically, `transformAtRuleContent` (the recursive handler used for `@media`/`@supports`/`@layer` children, lines 151-209) has **no `font-face` branch** — `@font-face` nested inside any conditional/layer at-rule is silently dropped by the web toolchain, never even reaching the wire format (worse than native Lynx, which at least encodes-then-discards it). `encodeCSS.ts` (`lynx-stack/packages/web-platform/web-core/ts/encode/encodeCSS.ts:100`) pushes a `Rule::new("FontFaceRule")` with raw declarations into the shared rkyv `RawStyleInfo` wire format (`RuleType::FontFace = 2`, already covered by lynx-vello's own `crates/lynx-template-decoder`, see `docs/web-binary-template.md`). At runtime, `StyleInfoDecoder::decode` (`lynx-stack/packages/web-platform/web-core/src/template/template_sections/style_info/style_info_decoder.rs:245`) regenerates literal `@font-face{ ... }` CSS text into a **separate `font_face_content` buffer** (kept apart from ordinary rules "because the font face should be placed at the head of the css content"), which `StyleSheetResource::new` (`style_sheet_resource.rs:49`) turns into its own `<style>` element (client) or raw string (SSR, `style_manager_server.rs:36`) injected into the page — i.e., **web-core delegates all `@font-face` semantics (src matching, unicode-range, font matching/fallback, network loading) to the real browser CSS engine**, doing no custom parsing, matching, or loading logic of its own. Consistent with this "let the browser do it" philosophy, **`lynx.addFont()` has no implementation anywhere in web-core** — no `addFont` method appears on any `NativeLynxProxy`/native-bridge implementation under `packages/web-platform/`; the only occurrences of the string `addFont` in that tree are inside bundled `lynx-core` JS chunks (the same shared JS library that defines the wrapper for all targets), meaning calling `lynx.addFont()` on the web target likely throws or no-ops rather than registering a font — **a functional gap lynx-vello must explicitly decide how to handle** (native-style `addFont` semantics don't exist to be "compatible" with on this target; ReactLynx apps calling it against web today get no working behavior to match).

### @supports Feature-Query Evaluation

Lynx has a fully-built, unit-tested `@supports` condition parser and evaluator (`lynx/core/renderer/css/ng/parser/supports_condition_parser.{h,cc}`, `lynx/core/renderer/css/ng/supports/supports_condition.{h,cc}`, `lynx/core/renderer/css/ng/supports/supports_evaluator.{h,cc}`) implementing CSS Conditional Rules Module Level 3/4 grammar: `not`, left-associative `and`/`or` chains (mixing without parens is a hard parse error, matching spec), parenthesized nesting, `(property: value [!important])` declaration checks, a catch-all "general enclosed" node that always evaluates `false` (spec-compliant forward-compat behavior), and a Lynx-only `-x-engine-version(begin[, end])` extension function for gating CSS on the host engine's version. Everything is exercised by `lynx/core/renderer/css/ng/parser/supports_condition_parser_test.cc` and `lynx/core/renderer/css/ng/supports/supports_evaluator_test.cc`, including exhaustive edge cases (uint16/uint32 overflow guards, `3.10` vs `3.1` precision, and/or mixing, general-enclosed fallback).

**Critical finding: this subsystem is architecturally complete but functionally dead code / not wired to any real stylesheet.** `ConditionRule::SetSupportsCondition` (`lynx/core/renderer/css/ng/style/condition_rule.h`) and `SharedCSSFragment::AddConditionRule` (`lynx/core/renderer/css/shared_css_fragment.cc:297`) are called **only from `rule_set_unittest.cc`** — grep across `core/` and `platform/` finds zero production call sites that parse a literal `@supports` at-rule out of a stylesheet and attach a `SupportsConditionNode` to a `ConditionRule`. Likewise, `SupportsEvaluator`'s constructor (`lynx/core/renderer/css/ng/supports/supports_evaluator.h`) is invoked nowhere in production code — only in its own test file. `RuleSet::MatchStyles` (`lynx/core/renderer/css/ng/style/rule_set.cc:68-91`) does correctly call `condition_rule->MatchesSupportsCondition(supports_evaluator)` and fails closed (treats as non-matching) if no evaluator is supplied or if the check returns false — so the *matching pipeline plumbing* is real, but nothing today produces a populated `SupportsEvaluator` or a `ConditionRule` with a real `@supports` condition from parsed CSS. This looks like an in-progress "ng" (next-gen) CSS rule engine that isn't yet the production path.

On the lynx-stack side, the offline build-time compiler (`lynx-stack/packages/tools/css-serializer/src/parse.ts:177-188`) does recognize `@supports` at-rule syntax and produces a `SupportsRule { prelude, rules }` AST node (`lynx-stack/packages/tools/css-serializer/src/types/LynxStyleNode.ts`), with `prelude` holding the raw unparsed condition text — there is no boolean evaluator on this side at all, not even a stub. Critically, grepping `lynx-stack/packages/tools/css-serializer/src/css/cssChunksToMap.ts` and `src/css/index.ts` shows **`SupportsRule`/`MediaRule`/`LayerRule` are never consumed downstream** — they're parsed into the AST and then dropped; nothing walks them into the final compiled style map. Consistent with this, `RawStyleInfo` (the rkyv wire struct this repo's decoder implements — see `lynx-vello/docs/web-binary-template.md`) has **no field whatsoever for conditional/`@media`/`@supports` rules**; `docs/web-binary-template.md` documents `css_id_to_style_sheet: HashMap<i32, StyleSheet>` with no conditional-rule variant. So today's actual `.web.bundle` artifacts that lynx-vello consumes cannot even encode a `@supports` block — any `@supports`-gated CSS an author writes is silently dropped by the lynx-stack web build pipeline before it ever reaches the runtime, regardless of what native Lynx (Android/iOS) might eventually do with the C++ engine described above.

**Implication for lynx-vello:** because the wire format has no representation for `@supports` at all, there is currently no bundle-level data for a stylo-based evaluator to act on — `@supports`-gated rules authored in ReactLynx are being silently dropped by the existing lynx-stack web toolchain, not evaluated in either direction. If lynx-vello wants ReactLynx-author compatibility with `@supports`-gated CSS (matching desktop-browser expectations, since that's what web-oriented ReactLynx authors will assume), it must (a) extend its own CSS-to-bundle compiler (or richer decoder) to actually preserve `@supports` conditions in the wire format, since neither existing pipeline does, and (b) implement a real evaluator. Given stylo already has genuine `@supports` support (it is a browser engine's CSS engine), the natural strategy is to let stylo's own supports-condition evaluator answer `(property: value)` declaration checks (arguably more spec-correct than Lynx's `UnitHandler`-based check, since it reflects the real render engine's parsing capability) while treating Lynx's `-x-engine-version()` extension as an additional custom function lynx-vello could optionally honor for compatibility with any Lynx-authored CSS relying on it (unlikely in practice since it's dead in both current runtimes). `selector()`/`font-tech()`/`font-format()` should ideally be evaluated for real (spec says so) rather than hardcoded false as Lynx's C++ evaluator does.
