# JS/Lepus runtime bindings, native modules & lifecycle

> Research: multi-agent sweep over `lynx/` and `lynx-stack/` (see [AGENTS.md](../../AGENTS.md) for the reference-repo shorthand and the W3C-first standards policy). Supersedes the earlier stub.

### Global JS API surface, native module bridge, and lifecycle

Lynx runs compiled ReactLynx output on **two logical JS contexts that share one `lynx` global shape but different implementations**: the **main thread (MTS, "Lepus"/`__LEPUS__`)**, which owns the Element tree and re-executes the render function synchronously on every update, and the **background thread (BTS)**, which hosts the "real" React/Preact component tree, event handlers, and business logic. Both are backed by the same C++ `App`/`LynxProxy`/`AppProxy` HostObjects (`lynx/core/runtime/js/bindings/js_app.{h,cc}`, `lynx/core/runtime/js/bindings/lynx.{h,cc}`) — the *set* of `lynx.*` and bare-global members differs by thread (see Tier/thread notes below), but there is no separate C++ class per thread; the JS-side `@lynx-js/types` package (`@lynx-js/types@3.7.0/types/{common,background-thread,main-thread}`) is the authoritative "what's available where" split and matches what ReactLynx (`lynx-stack/packages/react/runtime/src/snapshot/lynx/*`) actually wires up. Native module invocation is a third, orthogonal mechanism: JS calls `NativeModules.<Module>.<method>(...)`, which resolves through a `LynxModuleProviderFunction` to a `LynxModule` HostObject (`lynx/core/runtime/js/bindings/modules/lynx_module.{h,cc}`), and the module's C++/platform implementation replies via callbacks routed through `ModuleDelegate` (`lynx/core/runtime/js/bindings/modules/module_delegate.h`). Cross-thread/cross-context signaling (`lynx.getCoreContext()`, `getJSContext()`, `getUIContext()`, `getNative()`, `getDevtool()`, `getEngine()`) uses a separate `ContextProxy`/`MessageEvent` postMessage-style channel (`lynx/core/runtime/common/bindings/event/context_proxy.{h,cc}`, `message_event.{h,cc}`) with an internal wire vocabulary of `__On*`/`__*` event-type strings enumerated in `lynx/core/runtime/js/runtime_constant.h` — this is how lifecycle notifications (first screen, foreground/background, low memory, destroy) actually get delivered, wrapped for app code as `lynx.getJSModule('GlobalEventEmitter')` events or as the `tt` lifecycle hooks below.

### Lifecycle sequence and threading model

**Startup / first screen (cold start).** Native container decodes the `.web.bundle`/`.lynx.bundle`, constructs the MTS (`Lepus`) context first, and calls `App::LoadApp(...)` which invokes the compiled bundle's `loadCard` with the `AppProxy` `this`, `encodedData`, `initData`, `cacheData`, `cardConfig`, and `globalProps` (`lynx/core/runtime/js/bindings/js_app.cc:1997+`). On the ReactLynx side this surfaces as `globalThis.renderPage(data)` (`lynx-stack/packages/react/runtime/src/snapshot/lynx/calledByNative.ts:90`), which does `resetJSReady()`, sets `lynx.__initData`, creates the root page element (`__CreatePage`), runs `renderMainThread()` (a synchronous Preact-like render producing a `SnapshotInstance` tree and Element-PAPI calls), then flushes (`__FlushElementTree` implicit). If `__FIRST_SCREEN_SYNC_TIMING__ === 'immediately'`, `jsReady()` fires synchronously in the same tick; otherwise the MTS serializes its snapshot tree (`JSON.stringify(__root)`) and separately signals readiness once BTS calls back.

In parallel/subsequently, the BTS context is constructed, its `App`/`LynxProxy` set up, `injectTt()` wires `lynxCoreInject.tt.*` (`onLifecycleEvent`, `publishEvent`, `updateGlobalProps`, `updateCardData`, `onAppReload`, `callDestroyLifetimeFun`), and user code eventually calls `root.render(<App/>)` (`lynx-stack/packages/react/runtime/src/lynx.ts:87-109`). If BTS `root.render()` happens synchronously before native's "first screen" notification arrives, `lynx.getNativeApp().callLepusMethod(LifecycleConstant.jsReady, {})` tells native BTS is ready; otherwise the pending `firstScreen`/`__OnAppFirstScreen` lifecycle event (delivered via `__OnLifecycleEvent`/`ContextProxy` messaging, wire name `rLynxFirstScreen`) is queued (`delayLifecycleEvent`) until `root.render()` runs, then processed by `onLifecycleEventImpl('rLynxFirstScreen', ...)`: it re-runs Preact's internal `process()` flush, parses the MTS snapshot JSON, **hydrates** the BTS-side snapshot tree against it (reconciling event handler ids, the `jsReadyEventIdSwap` table), replays any events/lifecycle callbacks that arrived before hydration finished, computes a patch (`commitPatchUpdate`), and ships it back to MTS via `lynx.getNativeApp().callLepusMethod(LifecycleConstant.patchUpdate, obj, ...)` (wire name `rLynxChange`) so MTS applies the diff to the real element tree. This first-screen "MTS renders eagerly, BTS re-derives and hydrates/reconciles against it" pattern is the core of Lynx's dual-thread startup and must be replicated faithfully for behavioral compatibility, since app code's effects/refs/event bindings depend on hydration completing correctly.

**Steady-state updates.** BTS-driven `setState`/prop changes go through Preact's normal render → snapshot patch → `callLepusMethod('rLynxChange', ...)` → MTS applies patch via Element PAPI → `__FlushElementTree`. Native-driven data pushes (`updatePage`) go the other direction: MTS re-renders synchronously (unless `isJSReady` is false, in which case a full re-render+hydrate happens using the swap table) and flushes; then `tt.updateCardData` applies the same new data to BTS state.

**Foreground/background.** Native container (LynxView-equivalent) calls `LynxShell::OnEnterForeground()` / `OnEnterBackground()` (`lynx/core/shell/lynx_shell.cc:1292-1345`), gated by an `app_state_` dedup so redundant calls are no-ops. Each dispatches a `MessageEvent` of type `__OnAppEnterForeground`/`__OnAppEnterBackground` from `kCoreContext` to `kJSContext` (and, if the engine wants MTS notified, from `kNative` to `kCoreContext` as well). `App::Init()` installs listeners on the `kCoreContext` proxy that call `ResumeAnimationFrame()`/`PauseAnimationFrame()` on these events (`lynx/core/runtime/js/bindings/js_app.cc:1876-1883`) — i.e., **rAF is automatically paused while backgrounded**. There is no dedicated C++ "OnShow"/"OnHide" virtual in `core/renderer` — app-level show/hide is purely this MessageEvent/GlobalEventEmitter plumbing, so ReactLynx-level `onShow`/`onHide`-style hooks (if any exist in userland) are just `lynx.getJSModule('GlobalEventEmitter')` listeners the app registers itself, not engine callbacks we must implement as distinct virtual dispatch — confirmed absent from `lynx-stack/packages/react/runtime/src` search.

**Reload.** `lynx.reload(value, callback)` (BTS-callable) or a native-initiated reload triggers, on MTS, `reloadMainThread(data, options)`: bumps a reload-version counter, merges data into `lynx.__initData`, clears the snapshot-instance registry, builds a fresh root, re-runs `renderMainThread()`, hydrates old→new, flushes. On BTS, `tt.onAppReload` → `reloadBackground(updateData)`: tears down (`destroyBackground()`), bumps reload-version, merges `lynx.__initData`, and re-renders the same JSX tree fresh (`lynx-stack/packages/react/runtime/src/snapshot/lifecycle/reload.ts`).

**Destroy.** `tt.callDestroyLifetimeFun` (invoked by native, wire concept `__DestroyLifetime`/`App::CallDestroyLifetimeFun`) tears down: removes the "ctx not found" event listener, `destroyWorklet()`, `destroyBackground()`. `App`'s own destructor state machine tracks `kNotStarted → kStarted → kAppLoaded/kAppLoadFailed → kDestroying` (`lynx/core/runtime/js/bindings/js_app.h:341-348`), and `IsDestroying()` gates most `AppProxy`/`LynxProxy` methods against use-after-teardown.

**Other native→JS lifecycle signals** (all via the same `ContextProxy`/`MessageEvent` bus, wire names in `lynx/core/runtime/js/runtime_constant.h`): `__OnLowMemory`, `__OnComponentActivity`, `__OnSsrScriptReady`, `__OnComponentSelectorChanged`, `__ForceGcJSIObjectWrapper` (forces JSI object-wrapper GC — listened to per-`App::Init()`), and the full React-specific set (`__OnReactComponentRender`, `__OnReactCardDidUpdate`, `__OnReactComponentDidUpdate`, `__OnReactComponentDidCatch`, `__OnReactComponentCreated`, `__OnReactComponentUnmount`, `__OnComponentPropertiesChanged`, `__OnComponentDataSetChanged`) used by the (now largely legacy) Radon/class-component compat layer rather than Fiber/ReactLynx.

**Threading summary.** MTS and BTS are two independent JS realms (potentially on different native threads/via different JS engines, e.g. one on QuickJS/Lepus and one on V8/JSC), each with its own `Runtime`/`App`/HostObjects — they never share JS objects directly. All cross-realm communication is by serialized message passing: (1) the `tt`/`calledByNative` lifecycle-call surface (native "calls into" a realm by invoking a well-known global function), (2) the `ContextProxy`/`MessageEvent` bus (`postMessage`/`addEventListener` shape) for ad hoc typed messages including lifecycle notifications, and (3) the snapshot-patch commit protocol (`callLepusMethod('rLynxChange'/'rLynxJSReady'/...)`) specifically for reconciling the Fiber/ReactLynx tree between BTS's virtual tree and MTS's real Element tree. Native module (`NativeModules`) calls are logically synchronous-looking from JS but are dispatched through `ModuleDelegate::RunOnPlatformThread`/`RunOnJSThread`, so results return asynchronously via registered JS callbacks unless the specific module method is documented as sync.

### NodesRef.invoke()/fields() measurement API surface and native UIMethod catalog

**Architecture, not just an API list.** `NodesRef`/`SelectorQuery` is a dual-mechanism contract, and this matters for lynx-vello's design:

1. **Structural queries** (`select`/`selectAll`/`selectRoot`, `path()`, `fields()`) are resolved entirely off the DOM-adjacent tree — in the native engine by `PageProxy::GetFields`/`RadonNodeSelector`/`FiberElementSelector` (`lynx/core/renderer/page_proxy.cc`, `lynx/core/renderer/dom/fiber/fiber_node_info.cc`), and on web by the **wasm-compiled** `@lynx-js/lynx-core` runtime (ships as a prebuilt npm package, `@lynx-js/lynx-core@0.1.x`, output at `output/web/lynx_core.js` — this is almost certainly the same C++ selector/fields code compiled to wasm, not reimplemented in `web-core`'s TypeScript). **web-core's own TypeScript has no `fields()`/`path()` implementation at all** — grepping `lynx-stack/packages/web-platform/web-core/ts` for `GetFields`/`fields(` returns nothing outside `.d.ts`/type files. This is a real gap in what's inspectable from source for lynx-vello: the exact `fields()` wire contract is defined by the native `FiberNodeInfo::GetNodeInfo` C++ function, not by any TS/JS in this repo.
2. **UI-instance queries/mutations** (`invoke()` with `boundingClientRect`, `scrollTo`, `getScrollInfo`, etc.) require a live, mounted UI/DOM node and cross to the main thread. On web this is the single PAPI `__InvokeUIMethod` (`lynx-stack/packages/web-platform/web-core/ts/client/mainthread/elementAPIs/createInvokeUIMethod.ts`); on native (iOS) it is `PaintingContextDarwin::Invoke` → ObjC `LynxUIMethodProcessor invokeMethod:withParams:withResult:forUI:` (`lynx/core/renderer/ui_wrapper/painting/ios/painting_context_darwin.mm:675-736`, `lynx/platform/darwin/ios/lynx/base/LynxUIMethodProcessor.m`).

**The native UIMethod registry is NOT a fixed enum — it's an open, reflection-based per-class registry.** `LynxUIMethodProcessor` (`lynx/platform/darwin/ios/lynx/base/LynxUIMethodProcessor.m:40-107`) walks the Objective-C class hierarchy of the target `LynxUI` subclass at first use, collecting every method tagged with the `LYNX_UI_METHOD(name)` macro (`lynx/platform/darwin/ios/lynx/public/base/LynxUIMethodProcessor.h:18-23`) into a `{className: {methodName: block}}` cache, then looks up by class-walking to `LynxUI` base. web-core's `createInvokeUIMethod.ts:29-31` mirrors this exactly but with plain JS reflection: `typeof element[method] === 'function' ? element[method](params) : METHOD_NOT_FOUND` — i.e. **any public method on the resolved custom element is invocable by string name from ReactLynx JS**, with zero allowlisting. Only `"boundingClientRect"` is special-cased on web (computed via `getBoundingClientRect()` + `BoundingClientRectService` origin offset, not delegated to the element). This means lynx-vello's native engine needs an equivalent open/reflective (or explicit per-widget registration table) dispatch — a fixed enum will under-serve custom/host components that add their own methods.

**Error codes are shared** across native (`LynxGetUIResult`, `lynx/core/renderer/dom/lynx_get_ui_result.h:53-61`) and web (`ErrorCode`, `lynx-stack/packages/web-platform/web-core/ts/constants.ts:66-74`): `SUCCESS=0, UNKNOWN=1, NODE_NOT_FOUND=2, METHOD_NOT_FOUND=3, PARAM_INVALID=4, SELECTOR_NOT_SUPPORTED=5, NO_UI_FOR_NODE=6`; native additionally has `INVALID_STATE_ERROR=7, OPERATION_ERROR=8` which web never emits (web has no equivalent path to reach these, so `fields()`/`invoke()` failures on web can only ever surface 6 of the 9 native codes).

**Public TS surface** (from the actual shipped `@lynx-js/types@3.7.0` package, `background-thread/nodes-ref.d.ts` — this is the real public contract ReactLynx code is written against, not reconstructable from this repo's own source since `@lynx-js/types` isn't vendored):

```ts
interface BaseNodesRef {
  animate(animations): SelectorQuery;
  playAnimation(ids): SelectorQuery;
  pauseAnimation(ids): SelectorQuery;
  cancelAnimation(ids): SelectorQuery;
  setNativeProps(nativeProps): SelectorQuery;
}
interface NodesRef extends BaseNodesRef {
  invoke(options: { method, params?, success?, fail? }): SelectorQuery;
  path(callback): SelectorQuery;
  fields(fields: FieldsParams, callback): SelectorQuery;
}
```
`FieldsParams` = `{ id?, dataset?, tag?, index?, class?, attribute?, query? }`. Note: the TS type promises a `query: SelectorQuery` field, but the actual native `FiberNodeInfo::GetNodeInfo` switch (`lynx/core/renderer/dom/fiber/fiber_node_info.cc:28-83`) **has no `"query"` case** — only `id`, `dataset`/`dataSet`, `tag`, `unique_id`, `name`, `index`, `class`, `attribute` are handled. Requesting `query: true` in `fields()` is a documented-but-dead option; lynx-vello should not implement it as a real capability, or should treat it as always-absent from the result dict.

#### Cross-cutting notes for lynx-vello implementation

- **Dispatch mechanism choice**: native uses reflection over an `LYNX_UI_METHOD`-tagged Obj-C selector naming convention; web-core uses raw `element[method]` JS reflection. Both are effectively "any public method by string name," not a closed enum. lynx-vello (Rust) has no runtime reflection equivalent — the pragmatic path is an explicit per-widget trait/registry (e.g. a `UiMethod` trait object map keyed by `(widget_type, method_name)`), which is stricter than both references but necessary in Rust; this registry should be treated as the single place new UIMethods get added, mirroring the two catalogs above.
- **Async/threading model**: on native, `invoke()` runs on the main/UI thread and calls back asynchronously (`TemplateAssembler::LepusInvokeUIMethod`, `lynx/core/renderer/template_assembler.cc:1853-1927`, with an "enable pending task" mode that queues invokes on elements not yet created, via `element->AppendPendingInvokeTask`). This pending-task queuing behavior (invoke queued until the element mounts) needs an equivalent in lynx-vello if `invoke()` can race element creation — otherwise early `invoke()` calls will spuriously return `NODE_NOT_FOUND`/`UNKNOWN`.
- **fields()/path() vs invoke() have different consistency models**: `fields()`/`path()` are answered from the retained element/Radon tree — no live UI needed, works even off pending/unmounted async-created elements in some paths. `invoke()` (except structural fallthroughs) needs a live platform UI object (`findUIBySign:`/`NO_UI_FOR_NODE`), so it will fail immediately after a node is created but before paint. lynx-vello must decide, per API, whether it can answer from its retained DOM/style tree (`fields`-like) or must wait for a real vello scene node / stylo layout box to exist (`invoke`-like: `boundingClientRect`, `scrollIntoView`, screenshot, all scroll/list/text-selection methods).

### Global Error/Crash Boundary at the Native-Bridge Level

This is architectural, not a flat enumerable API surface, so it's described as a funnel diagram in prose plus a reference table. There are **three separate JS/Lepus realms** in Lynx, each with its own uncaught-exception funnel, and they do **not** feed into each other or into React error boundaries except at one narrow point (BTS `lynx.reportError`, which a component *can* call from inside a caught error, e.g. from `componentDidCatch`/`useErrorBoundary`, but that is a voluntary app-level call, not automatic propagation).

**1. BTS (background/full-JS-thread) realm — the only realm with a Preact tree.**
Every native→JS call in this realm goes through the JSI `Runtime::call`/`callAsConstructor` machinery (`lynx/core/runtime/js/jsi/quickjs/quickjs_helper.cc:155-204`, mirrored in `jsc_exception.cc`, `v8_exception.cc`, `jsvm_exception.cc`). After each such call, the engine also drains unhandled-promise-rejection queues (`LEPUS_MoveUnhandledRejectionToException`, `lynx/core/runtime/js/jsi/quickjs/quickjs_helper.cc:172,195`) — this is QuickJS's built-in analog of the browser's `unhandledrejection` event, and it feeds the *same* funnel as a thrown exception. Any exception that escapes this boundary (event-handler dispatch, `setTimeout`/`setInterval` callback, native-module/JSB callback invocation, `requireModule`, promise executor, etc.) is wrapped as a `JSIException` and passed to `Runtime::reportJSIException` (`lynx/core/runtime/js/jsi/jsi.cc:37-41`), which calls the single registered `JSRuntimeDelegate::OnJSIException` (`lynx/core/shell/runtime/bts/bts_runtime.cc:102-111`). `BTSRuntime::OnJSIException` (`lynx/core/shell/runtime/bts/bts_runtime.cc:1044-1060`) either (a) forwards to `App::OnAppJSError` if the JS app object is alive, which calls the JS SDK's `js_app.onAppError(_, error)` (`lynx/js_libraries/lynx-core/src/app/app.ts:865-867`) → `handleInternalError` → `handleError` → `reportError()` module function → `nativeApp.reportException(...)` — a JSI native-module binding (`lynx/core/runtime/js/bindings/js_app.cc:482-494,1795`) that is the terminal JS→native hop; or (b) if the app is gone/still initializing, builds a `base::LynxError` directly and calls `delegate_->OnErrorOccurred` (native-only path, no JS ever runs). Separately, `lynx.reportError(e)` (the public API, `lynx/js_libraries/lynx-core/src/lynx/lynx.ts:143-172`) is a **voluntary/manual** entry into the *same* `handleUserError`→`handleError`→`reportError()`→`nativeApp.reportException` pipeline — it is what all the `lynx.reportError(...)` call sites across `lynx-stack/packages/react/runtime/src/**` use as a last-resort catch-all inside their own local `try/catch`. So "automatic uncaught exception" and "manual lynx.reportError()" are two on-ramps into one shared native pipe, both surfacing at `base::LynxError` / `NativeFacade::ReportError` (`lynx/core/shell/native_facade.h:52`, implemented per-platform e.g. `lynx/core/shell/android/native_facade_android.cc:161`) — the actual host-app-visible crash/error callback (Android `LynxViewClient`-style delegate; iOS equivalent). Two JS-runtime-internal safety nets also feed this pipe pre-emptively: `wrapReport()` wraps every `setTimeout`/`setInterval` callback in a try/catch that calls `handleUserError` (`lynx/js_libraries/lynx-core/src/app/app.ts:886-905`) — the closest JS-land analog of `window.onerror` for timer callbacks — and the polyfilled/native `Promise`'s unhandled-rejection hook likewise calls `handleUserError` (`lynx/js_libraries/lynx-core/src/app/app.ts:907-933`), i.e. the `unhandledrejection` analog for BTS.

**2. MTS (main-thread-script / worklet) realm — no Preact tree, no error boundary possible.**
Main-thread functions (`'main thread'` worklets bound to touch/gesture/event handlers) run through `MTSRuntime`/Lepus `CallClosure` (native call site e.g. `lynx/core/renderer/events/touch_event_handler.cc:1436-1491`, entry symbol `runWorklet`) which on the JS side is `runWorkletImpl` (`lynx-stack/packages/react/runtime/src/worklet-runtime/workletRuntime.ts:80-102`) — **this function has no try/catch at all**; a throw inside a `main-thread:true` function propagates straight up through the native `CallClosure`/QuickJS call machinery and is caught only by the *same* generic JSI exception funnel described above (`Runtime::reportJSIException`, since `CallClosure` ultimately uses the same `LEPUS_Call`-family wrapper), landing at `MTSContext::ReportError`/`ReportErrorWithMsg` (`lynx/core/runtime/mts_context.cc:18-31`) → `MTSRuntime::ReportError` (`lynx/core/shell/runtime/mts/mts_runtime.cc:206-302`) → `delegate_->ReportError`. Because there is no Preact component tree executing on the main thread, **no `componentDidCatch`/`useErrorBoundary` can ever see an MTS-realm exception** — this is confirmed directly in code: ReactLynx's main-thread reimplementation of Preact hooks stubs `useErrorBoundary` to store the callback but return a permanent no-op reset function (`lynx-stack/packages/react/runtime/src/core/hooks/mainThreadImpl.ts:180-184`), i.e. the hook exists in the main-thread hook table for API-shape compatibility but cannot actually recover/re-render on error. Snapshot-patch application on the main thread (`updateMainThread`, `lynx-stack/packages/react/runtime/src/snapshot/lifecycle/patch/updateMainThread.ts:68-84`, and its Element-Template twin `lynx-stack/packages/react/runtime/src/element-template/native/patch-listener.ts:73-82`) likewise wraps the patch-apply loop in only a `finally` (for timing marks), not a `catch` — an exception during patch application is uncaught JS-side and falls to the same native funnel. The one exception: **delayed `runOnMainThread` task dispatch** (`updateMainThread.ts:88-96`, `patch-listener.ts:89-98`) and **`renderMainThread`'s initial render** (`lynx-stack/packages/react/runtime/src/snapshot/lifecycle/render.ts:14-34`, `.../element-template/runtime/render/render-main-thread.ts:36-46`) *do* wrap in `try { … } catch (e) { lynx.reportError(e) }` — but `lynx.reportError` on the MTS/LEPUS side is bound differently: `lynx-stack/packages/react/runtime/src/snapshot/lynx/env.ts:22-27` shows that under `__LEPUS__` (main-thread realm), `lynx.reportError` is redefined to call the native `_ReportError(error, {errorCode: 1101 /* LYNX_ERROR_CODE_LEPUS */})` directly — bypassing the BTS JS-SDK's `handleError`/`Reporter` pipeline entirely and going straight to native. This is a materially different code path from the BTS `lynx.reportError`, despite the identical call syntax used throughout the shared `react/runtime` sources.

**3. Native-module (JSB) invocation realm.** Native-module callback invocation (`lynx/core/runtime/js/bindings/modules/lynx_jsi_module.cc`, `lynx_module.cc`) has no explicit try/catch of its own either; it relies on the same generic `Runtime::call` JSI wrapper for exception capture. `ModuleCallback::ReportLynxErrors` (`lynx/core/shell/runtime/bts/bts_runtime.cc:520`) additionally surfaces module-specific errors (e.g. malformed callback args) through the delegate's error-reporting path independent of a thrown-JS-exception.

**How this composes with React error boundaries.** `useErrorBoundary`/`componentDidCatch` (BTS realm only) are pure re-exports/subclass hooks from `preact`/`preact/compat` — `lynx-stack/packages/react/runtime/src/core/hooks/react.ts:8,170` re-exports `useErrorBoundary` from `preact/hooks` completely unwrapped (no ReactLynx-specific instrumentation), and no file under `lynx-stack/packages/react/runtime/src/` overrides Preact's internal `options._catchError`/`options.__e` mechanism (confirmed by grep — the only `componentDidCatch`/`useErrorBoundary`/`ErrorBoundary` hits in `packages/react/` are re-exports, type declarations, and tests, not custom implementations). This means: React error boundaries catch **synchronous throws during BTS-realm render/commit of the same-thread component subtree only** (standard Preact behavior — throws during `render()`, lifecycle methods, and effect callbacks of descendant components). They do **not** catch: (a) MTS/main-thread-script worklet exceptions (different realm/thread, no Preact tree there at all), (b) exceptions thrown inside native-module (JSB) async callbacks that fire outside the render/commit call stack (these hit the JSI funnel and `lynx.reportError`/`OnAppJSError` directly, never entering Preact's diff/render, so `componentDidCatch` is architecturally unreachable for them), (c) exceptions in `lynx.reportError()`'s own call sites — these are catch-alls that fire *after* the local `try/catch`, i.e. "last resort" reporting that intentionally happens *outside* any component tree and is therefore also invisible to error boundaries. In short: **error boundaries are a leaf-level, same-realm, synchronous-render-only mechanism; `lynx.reportError`/native `OnJSIException`/`OnErrorOccurred` is the orthogonal, cross-realm, always-on crash/error funnel that a component (in a `catch` block or `componentDidCatch` handler) may optionally feed into, but which also runs completely independently for every kind of exception a React error boundary structurally cannot see.**

### Soft-Keyboard Visibility Events & Keyboard-Avoiding Layout

This is a platform-behavior area with **no lynx-stack/web equivalent whatsoever** — confirmed by grep across `lynx-stack/packages/web-platform` and `lynx-stack/packages/react` source trees (only unrelated DOM `KeyboardEvent` — the keypress type — appears, in `lynx-stack/packages/web-platform/web-elements/src/elements/XTextarea/XTextareaAttributes.ts` and test files). The web target's iframe-based MTS realm has no analogous concept: browsers don't expose IME/soft-keyboard geometry to embedded content in a way `web-core` taps into, so lynx-vello has zero prior art to port from lynx-stack and must design this from scratch against the native (Android/iOS) engine's actual behavior in `lynx/`.

This is architectural rather than a flat enumerable list, so a summary of the mechanism precedes the table.

**How it works in native Lynx (both platforms share the same conceptual model, independently implemented):**

1. **Detection.** Each platform observes system-level layout/inset changes to infer keyboard show/hide and height:
   - Android (`lynx/platform/android/lynx_android/src/main/java/com/lynx/tasm/behavior/KeyboardEvent.java`, `KeyboardMonitor.java`): a transparent, non-focusable, non-touchable `Dialog` (`KeyboardMonitor`) is created with `SOFT_INPUT_ADJUST_RESIZE`, layered over the activity purely so its decor view's `ViewTreeObserver.OnGlobalLayoutListener` fires on resize; comparing visible-frame height against a cached full height (with `KEYBOARD_LOWER_THRESHOLD`/`KEYBOARD_HIGHER_THRESHOLD` ratio heuristics to filter rotation/multi-window noise) yields a `visible` boolean and `keyboardHeight`/`keyboardHeightCompat` in dp. Android 11+ can instead use `WindowInsetsAnimation.Callback` on the IME inset type for animation-synced tracking (`KeyboardInsetsAnimationDispatcher`, R+ only).
   - iOS (`lynx/platform/darwin/ios/lynx/event/LynxKeyboardEventDispatcher.m`): subscribes to system `UIKeyboardWillShowNotification`/`UIKeyboardWillHideNotification`, reading `UIKeyboardFrameEndUserInfoKey` for height and `UIKeyboardAnimationDurationUserInfoKey`/`AnimationCurveUserInfoKey` to mirror the system's show/hide animation timing/curve exactly (with an iOS-version-gated fallback that walks the window hierarchy for `UIRemoteKeyboardWindow`/`UIInputSetHostView` pre-iOS16).

2. **Global JS event.** Both platforms fire a Lynx **global custom event** named `keyboardstatuschanged` (`LynxKeyboardEvent.KEYBOARD_STATUS_CHANGED` in `lynx/platform/android/lynx_android/src/main/java/com/lynx/tasm/event/LynxKeyboardEvent.java`; `KEYBOARD_STATUS_CHANGED` macro in the iOS dispatcher) via `sendGlobalEvent`/`sendGlobalEvent:withParams:`, with args `["on"|"off", height, heightCompat]` (Android) or `["on"|"off", height]` (iOS). This is a page-global event apps subscribe to via the standard Lynx global-event-listener API (`lynx.getJSModule('GlobalEventEmitter').addListener('keyboardstatuschanged', ...)` pattern used elsewhere in the engine) — this is distinct/separate from the per-element composition events already covered under form-input/IME composition.

3. **Built-in keyboard-avoiding layout ("avoid-keyboard").** Independent of the JS event, native input elements have first-class avoid-behavior wired directly into the renderer, driven by two element props:
   - `avoid-keyboard` (bool) — Android: `LynxProp` in `lynx/platform/android/lynx_xelement/lynx_xelement_input/src/main/java/com/lynx/xelement/input/LynxUIBaseInput.kt:304`; iOS: `LYNX_PROP_SETTER("avoid-keyboard", ...)` in `lynx/platform/darwin/ios/lynx_xelement/input/LynxUIBaseInput.m:381`.
   - `avoid-keyboard-spacing` (length) — extra gap kept between the focused input's bottom edge and the keyboard's top edge — Android at `LynxUIBaseInput.kt:314`; iOS `LynxUIBaseInput.m:385`.
   
   When the focused input has `avoid-keyboard=true`, on keyboard show/hide/resize the engine computes an "avoid distance" (how far to translate the LynxView/root) from the input's on-screen bottom position, the keyboard top, and the spacing prop, then animates `translationY` (Android, `View.animate()`, `KEYBOARD_ANIMATION_DURATION_MS = 285ms`, `PathInterpolator(0.2,0,0,1)`) or the root view's `frame.origin.y` (iOS, `UIView animateWithDuration:` using the system's actual keyboard animation curve/duration) on the **whole LynxView/root**, not just the input — i.e., it shifts the entire page up, not just scrolls the input into view. This is implemented as a dedicated state machine (`KeyboardAvoidingContext` on Android, `LynxKeyboardAvoidingContext` on iOS) that tracks an "active target" (the currently-focused avoid-eligible input), reacts to focus-change (`OnGlobalFocusChangeListener`/`isFirstResponder`), to prop changes, to input relayout, and to keyboard show/hide, resetting the translation to 0 when the input blurs or keyboard fully hides. Android additionally special-cases IME inset *animation* frames (API 30+/R) to keep the avoid-distance visually in sync frame-by-frame with the system keyboard slide animation, with timer-based fallbacks (`INSETS_ANIMATION_START_FALLBACK_DELAY_MS = 64ms`, `+80ms` end padding) if the animation callback never fires.

4. **`android-set-soft-input-mode` prop.** Android-only per-input override of the window's `windowSoftInputMode` behavior — values `"unspecified" | "nothing" | "pan" | "resize"` map 1:1 to `SOFT_INPUT_ADJUST_*` constants (`LynxUIBaseInput.kt:629`), letting an app opt an input out of the `avoid-keyboard` JS-driven translation in favor of the OS's native resize/pan handling.

5. **Config flag `keyboardCallbackPassRelativeHeight`** (`useRelativeKeyboardHeightApi()` in `lynx/platform/android/lynx_android/src/main/java/com/lynx/tasm/PageConfig.java`, defined in `lynx/core/template_bundle/template_codec/binary_decoder/lynx_config.yml:1214`) — Android-only page-config bit (default `true` since SDK 2.2) controlling whether keyboard-height callbacks report height relative to the LynxView's own bottom (correcting for decor-view offset in multi-window/split-screen) vs. legacy absolute decor-view-relative height. When enabled, `LynxView.onLayout` (`lynx/platform/android/lynx_android/src/main/java/com/lynx/tasm/LynxView.java:1174`) additionally re-triggers keyboard-change detection on every layout pass, not just on keyboard show/hide.

**Key behavioral facts for lynx-vello's design (no direct code to port, but must replicate behavior):**
- Detection is heuristic/polyfilled on both platforms (Android via an invisible resizable dialog + visible-frame ratio thresholds; iOS via system notifications) because neither OS has a first-class "keyboard visibility" event — a native cross-platform engine (e.g. via `winit`/platform windowing) will need its own analogous detection per target platform (Android: `WindowInsets`/IME callback; iOS: `UIKeyboardWillShowNotification` equivalent; desktop: largely N/A, no soft keyboard).
- The engine moves the **whole page/root view**, not an individual scroll container, to avoid the keyboard — this is a translation of the render tree's root layer, not a CSS-level resize of the viewport or a `scrollIntoView`.
- `keyboardstatuschanged` is a **global event**, delivered through the same "global custom event" channel as other page-global signals (screen metrics change, etc.) — lynx-vello's runtime needs a global-event bus with this exact name/payload shape to be drop-in compatible with existing ReactLynx app code that listens for it.
- Height units: Android emits dp (`height`/`heightCompat` divided by `mDpi`); iOS emits points (native UIKit height, already resolution-independent). No px/CSS-pixel conversion quirk beyond the standard density normalization already used elsewhere in the engine.

### Screen/Device Metrics & Pixel-Density JS API Contract (SystemInfo)

`SystemInfo` is a plain global object (not a class/module import) injected into every JS/Lepus realm — background thread, main-thread/Lepus (MTS), and worklet-runtime — that ReactLynx apps read synchronously (`SystemInfo.pixelRatio`, `SystemInfo.platform`, etc.) for JS-side responsive logic, distinct from CSS media queries. #### Population, scoping, and live-update mechanics (architectural — not a clean per-row enumeration)

**Native (Android/iOS):** `SystemInfo` is backed by a single process-wide C++ singleton, `tasm::Config` (`lynx/core/renderer/tasm/config.h`/`.cc`), populated by `Config::InitPixelValues(width, height, ratio)` and `Config::InitializeVersion(osVersion)`. On Android this is invoked from a JNI entry point `UpdateDevice` (`lynx/core/renderer/utils/android/device_display_info.cc:20-25`) called by `DisplayMetricsHolder.setScreenDisplayMetrics` (`lynx/platform/android/.../DisplayMetricsHolder.java:244-253`); on iOS from `LynxEnv.initLayoutConfig:` (`lynx/platform/darwin/common/lynx/LynxEnv.mm:465-470`), invoked once from `LynxTemplateRender setUpEnvWidthScreenSize:` (`lynx/platform/darwin/ios/lynx/LynxTemplateRender.mm:336-339`) during view setup — **not** re-invoked on rotation. `Config::Platform()` is resolved at native compile time via one of five `sysinfo_*.cc` translation units selected by build target (`lynx/core/renderer/tasm/sysinfo/sysinfo_{android,ios,harmony,headless,default}.cc`), so "platform" is a compile-time constant, not a runtime OS query.

Android *does* expose a live per-`LynxView` metrics-refresh path — `LynxView.updateScreenMetrics(width, height)` (`lynx/platform/android/lynx_android/src/main/java/com/lynx/tasm/LynxView.java:437-444`) → `LynxTemplateRender.updateScreenMetrics` (`lynx/platform/android/.../LynxTemplateRender.java:796-811`) → native `ElementManager::UpdateScreenMetrics` (`lynx/core/renderer/dom/element_manager.cc:669-684`). However, this path only refreshes `LynxEnvConfig`'s screen size for **CSS unit computation** (`vw`/`vh`/`rpx`/screen-relative values, flagged via `DynamicCSSStylesManager::kUpdateScreenMetrics` in `lynx/core/renderer/css/dynamic_css_styles_manager.h:80`) — it does **not** touch the `Config` singleton that backs JS-visible `SystemInfo.pixelWidth/pixelHeight`, and does not dispatch any JS-observable event. Grepping all of `lynx/platform/android` for calls to this API found only DevTool/CDP device-emulation and screen-recording tooling (`LynxDevtool.java:298`, `LynxInspectorOwner.java:419`, `LynxRecorderActivity.java:153,159`, `LynxRecorderView.java:47`) — i.e., in production, no OS `onConfigurationChanged`/rotation callback wires into this at all; host apps would have to do so themselves, and even then only CSS units react, not the `SystemInfo` JS global. Separately, `DisplayMetricsHolder.updateOrInitDisplayMetrics` (`lynx/platform/android/.../DisplayMetricsHolder.java:80-134,161-175`) *does* detect real device orientation/density changes and re-invokes the JNI `UpdateDevice`/`Config::InitPixelValues` path — but this updates the shared process-global `Config` singleton (explicitly documented in that file as legacy/deprecated/"polluted" shared state that individual LynxViews should not rely on), not a per-view live value, and still triggers no JS event.

Theme is the one field with a live-update story, but it rides the heavyweight generic config-update path: `RadonPage::UpdateConfig`/`RadonComponent::UpdateSystemInfo`/`RadonLazyComponent::UpdateSystemInfo` (`lynx/core/renderer/dom/vdom/radon/radon_page.cc:794-802`, `radon_component.cc:70,656`, `radon_lazy_component.cc:92,138`) recompute `GenerateSystemInfo(&config)` whenever `updateGlobalProps`/page-config updates fire (Android: `LynxView.updateGlobalProps` → `LynxTemplateRender.java:3866` `nativeUpdateGlobalProps`), triggering a full `$renderPage` re-render (`to_refresh=true` branch), not a targeted "systeminfo changed" subscription.

**Web target (`lynx-stack/packages/web-platform/web-core`):** `createSystemInfo()` (`ts/client/mainthread/LynxViewInstance.ts:51-62`) builds the object once per `LynxViewInstance` construction by spreading `systemInfoBase` (`platform: 'web'`, `lynxSdkVersion: '3.0'`, from `ts/constants.ts:31-34`) with module-load-time `pixelRatio`/`pixelWidth`/`pixelHeight` (captured from `window.devicePixelRatio` / `window.screen.availWidth|availHeight` at `ts/client/mainthread/LynxViewInstance.ts:48-50`, i.e., **once per page load of the host page**, shared across all `LynxView` instances on that page) and an optional `browserConfig` override (settable via the `<lynx-view browser-config="...">` attribute, `ts/client/mainthread/LynxView.ts:24-29,113-131`, letting a host app manually inject arbitrary `pixelRatio`/`pixelWidth`/`pixelHeight`/custom keys — this is the only override/customization point that exists). The frozen object (`Object.freeze`) is exposed to MTS/Lepus directly (`createMainThreadGlobalAPIs.ts:49,80`), and to BTS (the worker) by one-time postMessage in `WorkerStartMessage.systemInfo` (`types/WorkerStartMessage.ts:11`, sent from `Background.ts:160`, received once in `client/background/index.ts:13-15` — `if (!globalThis.SystemInfo) globalThis.SystemInfo = message.systemInfo`). SSR (`server/createServerLynx.ts:53-55`) supplies only `{ platform: 'web-ssr' }` — no pixel metrics at all server-side. No `resize`/`orientationchange`/`matchMedia` listener anywhere in `web-core` ever re-derives or re-sends `SystemInfo`; a browser window resize post-load is invisible to `SystemInfo` (though CSS media queries/`vw`/`vh` in the same page *would* react live via the browser's own CSS engine — a real behavioral split between the CSS layer and the `SystemInfo` JS-global layer that lynx-vello must reproduce or consciously fix).

E2e fixtures confirm the consumer contract: `SystemInfo.platform`/`pixelRatio`/`pixelHeight`/`pixelWidth` are read synchronously in a React `useEffect` (BTS) in `lynx-stack/packages/web-platform/web-core-e2e/tests/reactlynx/api-SystemInfo/index.jsx` and `.../api-SystemInfo-height-width/index.jsx`, and also read synchronously inside a `main-thread:bindTap` worklet (MTS realm) in `.../tests/reactlynx/basic-mts-systeminfo/index.jsx` — confirming `SystemInfo` must be present, synchronously, in both thread realms from the first tick of app code.

#### Implications & recommendations for lynx-vello

- **Reproduce the "static snapshot" contract for ReactLynx compatibility**: apps written against real Lynx do not expect `SystemInfo.pixelWidth`/`pixelHeight`/`pixelRatio` to update live and do not commonly subscribe to a change event (none exists) — so lynx-vello's `SystemInfo` should also be a one-time-computed, frozen-at-view-creation object to match observed behavior, not a live-binding reactive object (which would technically be more "correct" but would diverge from what ReactLynx code paths assume/were tested against, e.g. list-rendering version gates keyed to a stable `lynxSdkVersion` string).
- **Do not centralize metrics in a process-global singleton** — this is called out as a known bug/tech-debt in the native codebase itself (`DisplayMetricsHolder.java` doc comment: "Ideally each lynx view should have its own sandboxed world... do NOT use the global screen metrics here"). lynx-vello should scope `SystemInfo` per rendering surface/window from the start.
- **Pick a stable, high `lynxSdkVersion`/`engineVersion`** so that ReactLynx runtime code with internal version gates (list diffing bugfixes, etc.) takes its modern/fixed behavior path — this is a hard compatibility requirement discovered via `listUpdateInfo.ts:207-211`, not an optional nicety.
- **`theme` should be treated as an opaque app-controlled JSON blob**, populated from whatever config/global-props channel lynx-vello uses for app-level config, updated only via a full config-refresh (matching native), not a dedicated push/subscribe theme API — there is no native precedent for the latter.
- **Gap to flag explicitly (not a Lynx-vs-W3C quirk, but worth deciding deliberately)**: real Lynx has *no* mechanism at all for JS code to be notified of a DPI/orientation change via `SystemInfo` — this is arguably a shortfall versus the W3C web platform (which has live `window.devicePixelRatio` change detection via `matchMedia('(resolution: ...)')` listeners, and `resize`/`orientationchange` events). Since this project's mandate is to intentionally follow W3C over Lynx quirks only where Lynx *diverges from a standard Lynx claims to implement* (not simply where Lynx lacks a feature entirely), this is presented as a design choice rather than a mandated deviation: lynx-vello could optionally add a live DPI-change/resize event as an *additive* extension without breaking `SystemInfo`'s snapshot contract, e.g. by only firing it into the CSS engine (media query re-evaluation, already expected to be live) while leaving the `SystemInfo` JS global static, exactly mirroring the real split observed above between CSS-vw/vh-live-refresh (`UpdateScreenMetrics`) and JS-`SystemInfo`-static.
