# Built-in components (web-elements ↔ Lynx JSX tags)

> Research: multi-agent sweep over `lynx/` and `lynx-stack/` (see [AGENTS.md](../../AGENTS.md) for the reference-repo shorthand and the W3C-first standards policy). Supersedes the earlier stub.

### Built-in Components (web-elements custom-element inventory)

Scope: every custom element registered under `lynx-stack/packages/web-platform/web-elements/src/elements/**` via the `@Component(tagName, mixins[], template?)` decorator (`lynx-stack/packages/web-platform/web-elements/src/element-reactive/component.ts:348` — decorator ultimately calls `customElements.define(tag, CustomElement)`), plus the JSX-tag → custom-element-tag mapping used by the mainthread element API. Confirmed authoritative mapping in `lynx-stack/packages/web-platform/web-core/ts/constants.ts:76-92` (`LYNX_TAG_TO_HTML_TAG_MAP`) and its Rust mirror `lynx-stack/packages/web-platform/web-core/src/constants.rs`; actually consumed in `lynx-stack/packages/web-platform/web-core/ts/client/mainthread/elementAPIs/createElementAPI.ts:226-229` via `document.createElement(LYNX_TAG_TO_HTML_TAG_MAP[tagName] ?? tagName)` — i.e. tags absent from the map (e.g. `x-swiper`, `x-input-ng`, `x-textarea`, `x-audio-tt`, `x-foldview-ng`, `x-refresh-view`, `x-overlay-ng`, `x-viewpager-ng`, `x-blur-view`) pass through **verbatim** as the JSX tag name, which is also their registered custom-element tag. These pass-through tags are also the ones lazily/dynamically loaded per `LYNX_TAG_TO_DYNAMIC_LOAD_TAG_ID` (`lynx-stack/packages/web-platform/web-core/ts/constants.ts:107-130`).

**Background quirk that applies to every component below** (not repeated per row): all Lynx elements default to `--lynx-display: linear` / `--lynx-linear-orientation: vertical` (a proprietary flex-like box model implemented via CSS custom properties + `@container style()` queries), not W3C `display: block`. See `lynx-stack/packages/web-platform/web-core/css/index.css:47-51,107-111` and the `LinearContainer` reactive mixin (`lynx-stack/packages/web-platform/web-elements/src/compat/LinearContainer/LinearContainer.ts`). **Deviation**: real CSS/W3C default is `display: block` (or `inline` for text-level content) with static/normal-flow layout. lynx-vello should model Lynx's actual default (linear/vertical-flex) since that is the behavior ReactLynx apps depend on — flag this as a documented intentional Lynx-quirk-preserving default, distinct from the z-index divergence which we deliberately do NOT preserve.

### Form Input Constraint-Validation & IME Composition Contract (x-input / x-textarea)

Confirmed by direct source inspection: **the HTML5 constraint-validation model (`required`, `pattern`+`checkValidity`/`reportValidity`, `min`/`max`, `ValidityState`, `setCustomValidity`, `<form novalidate>`) is entirely absent** from both `x-input` and `x-textarea` in lynx-stack's web-platform implementation, and there is no equivalent in the native LynxJS engine either — `grep` for `checkValidity|reportValidity|ValidityState|willValidate|setCustomValidity|novalidate` across `lynx-stack/packages/web-platform` and `lynx-stack/packages/react` (source, not `dist/`) returns zero hits, and the same search over `lynx/` (C++ engine) also returns zero hits. The shadow-DOM `<form id="form" method="dialog">` wrapper (`lynx-stack/packages/web-platform/web-elements/src/elements/htmlTemplates.ts:86-95,375`) exists only as a plumbing device to capture native `submit`/`blur`/`focus` events and re-dispatch them as Lynx bridge events (`bindconfirm`/`bindblur`/`bindfocus` via `renameEvent`) — it is never actually submitted by a user action (its CSS sets `display: none`), so the browser's native constraint-validation-on-submit gate (which would normally block submission and show validation bubbles) is never exercised. `pattern` is exposed only as a custom regex **input filter** (`x-input`'s `setInputFilter({pattern})` sets the native `pattern` attribute purely so browsers can apply it as an `attr` CSS/behavior hook — it is not read back via `checkValidity()` anywhere) and as the `input-filter` attribute that strips matching characters live on every `input`/`compositionend` event (`XInputEvents.ts:124-134`, `XTextareaEvents.ts:78-81,104-107`) — i.e. Lynx's "pattern" is a **live sanitizing filter**, not a post-hoc validity check. There is no `min`/`max`/`step` numeric-range constraint at all (only `type="number"` triggers a client-side character-class filter via `#numberInputFilter`, `XInputEvents.ts:22,126-128`); `step="any"` is hardcoded in the template but never read. `maxlength` is the only real constraint enforced, and it is enforced by truncation (both on the `value` attribute setter and via the native HTML `maxlength` attribute), not by a validity flag. Required-ness (`required` attribute) is not observed by either `XInputAttribute`/`InputBaseAttributes` or `XTextareaAttributes`/`TextareaBaseAttributes` (`observedAttributes` lists checked directly: `XInputAttribute.ts:19`, `InputBaseAttributes.ts:20-27`, `XTextareaAttributes.ts:17-23`, `TextareaBaseAttributes.ts:16-24` — none list `required`, `pattern`, `min`, `max`, or `minlength`). ReactLynx (`lynx-stack/packages/react`) adds no framework-level validation API over these either (zero hits for the same grep, and zero references to `XInput`/`XTextarea`/`x-input`/`x-textarea` in `packages/react/*/src`), so there is no "React-controlled-input" validation contract to preserve beyond what's below.

On IME composition: the DOM-standard three-event contract (`compositionstart` → `compositionupdate`* → `compositionend`) is **not surfaced to Lynx's JS runtime at all**. The web-elements shim wires only a single native listener, `compositionend` (`XInputEvents.ts:44-47`, `XTextareaEvents.ts:43-46`), which — together with the native `input` event's `event.isComposing` flag (`XInputEvents.ts:80`, `XTextareaEvents.ts:82`) — is used purely to gate the *outgoing* bridge event `lynxinput`: while composing, `lynxinput` is suppressed unless `send-composing-input` is set (`#sendComposingInput`, `XInputEvents.ts:60-63,82`, `XTextareaEvents.ts:59-62,84`); on `compositionend` a final `lynxinput` fires with `isComposing:false` regardless. No `compositionstart`/`compositionupdate` are ever listened for, dispatched, or exposed as Lynx events — there is no way for a ReactLynx `bindcompositionstart`/`bindcompositionupdate` prop to ever fire on `x-input`/`x-textarea` in web-platform, and no such event names appear anywhere in `lynx-stack` (all composition-event hits are internal browser-native listeners, not re-dispatched Lynx events). Confirmed no native Android/iOS analogue either: `lynx/platform/android/.../LynxInputConnectionWrapper.kt` overrides Android's own `InputConnection.setComposingText`/`setComposingRegion` (the IME composing-text protocol, structurally different from and not translated into DOM composition events), and `lynx/platform/embedder/plugin/cef/.../osr_ime_handler_win.cc` is unrelated embedder-chrome (CEF off-screen-rendering window) IME plumbing, not element-level composition events — confirming there is no cross-platform "Lynx composition event" concept to port, only this web-platform-specific `compositionend`-triggered flush behavior.

For lynx-vello, since the goal is ReactLynx *behavioral* compatibility (not W3C purity) for the JS-visible contract, and Lynx's own contract already deliberately omits `compositionstart`/`compositionupdate` and the whole constraint-validation model, lynx-vello should replicate the **absence** faithfully (do not invent `checkValidity`/`compositionstart` support that ReactLynx apps can't rely on) while implementing its own native IME/composition handling (via winit/native text-input APIs) using the same suppress-during-compose-then-flush-on-end semantics as the reference, so `bindinput` timing matches pixel-for-pixel what a real Lynx app observes today.

### Dynamic/Lazy Component Loading (`<component is="...">`, LazyBundle)

**Architecture summary (Radon/legacy engine, the ground truth read here).** A dynamic component is not a distinct DOM primitive to the compiler — `<component is="foo">` compiles to a `RadonLazyComponent` node (`lynx/core/renderer/dom/vdom/radon/radon_lazy_component.h/.cc`), a `RadonComponent` subclass that starts **empty** (no mould, no context, no style sheet) and is registered in `PageProxy::empty_component_map_` (`lynx/core/renderer/page_proxy.cc:905-957`) while its bundle is outstanding. Loading is driven by a dedicated subsystem in `core/resource/lazy_bundle/` (`AGENTS.md` there literally calls it out as its own module: "lazy-bundle loading support: request description, lifecycle options, loader orchestration, and lazy-bundle utilities" — `lynx/core/resource/lazy_bundle/AGENTS.md`), separate both from the single-entry bundle decode this repo already does and from the static built-in-element inventory.

Full lifecycle, traced end to end:
1. **Resolve URL** — `TemplateAssembler::RequireTemplateEntry` (`lynx/core/renderer/template_assembler.cc:2728`) looks up an existing `TemplateEntry` by name; if absent, checks `preload_template_bundles_` (bundles pushed ahead of time via `registerDynamicComponent`/`InsertLynxTemplateBundle`, `lynx/core/renderer/template_entry_holder.cc:74-95`); only then does it fall through to a real fetch.
2. **Fetch** — `LazyBundleLoader::RequireTemplate` (`lynx/core/resource/lazy_bundle/lazy_bundle_loader.cc:178`) calls the platform-injected `pub::LynxResourceLoader::LoadResource` with `LynxResourceType::kLazyBundle`; this is the actual network/asset-loading hook a host app implements (e.g., HTTP fetch, bundled asset lookup). `LazyBundleLoader::FetchBundle`/`DidFetchBundle` is a parallel path used by the JS `fetchBundle`/preload APIs.
3. **Decode** — `DecodeBundle` (file-local helper in `lazy_bundle_loader.cc:30`) runs `LynxTemplateBundle::FromBinaryGreedy` on the raw bytes — this is the exact `.web.bundle`/`.lynx.bundle` decoder this repo already implements for a single entry, invoked again per lazy bundle with its own `TemplateEntry`.
4. **Build entry / splice into tree** — `TemplateAssembler::LoadComponentWithCallbackInfo` (`lynx/core/renderer/template_assembler.cc:1406`) builds a brand-new `TemplateEntry` for the URL (own Lepus/JS VM context, own `CSSStyleSheetManager`, own component moulds) via `BuildComponentEntryInternal`, then `RadonLazyComponent::SetContext`/`LoadLazyBundleInternal` (`radon_lazy_component.cc:141-224`) attaches that entry's mould/style-sheet/VM to the (previously empty) node and does a full first render + Radon diff/dispatch (`DispatchForRender`, `PreRender`, `DispatchForDiff`) to graft the loaded subtree under the existing parent page's tree.
5. **Resolve loading state / fire events** — `PageProxy::OnLazyBundleLoadedSuccess`/`OnLazyBundleLoadedFailed` (`lynx/core/renderer/page_proxy.cc:335-396`) look the node up in `empty_component_map_` by uid, then either finish loading or call `RadonLazyComponent::RenderFallback()`. Success/failure is also surfaced as a `bindEvent`/`catchEvent` named `"success"`/`"fail"` (constants in `lynx/core/resource/lazy_bundle/lazy_bundle_utils.h:17-18`) dispatched via `TemplateAssembler::SendLazyBundleBindEvent` → `TouchEventHandler::HandleCustomEvent` (`lynx/core/renderer/template_assembler.cc:3247`), i.e. the same declarative-event-binding machinery as any other DOM event, not a special API. A legacy `GlobalEventEmitter.emit('onDynamicComponentEvent', ...)` fallback fires **only on failure**, for backward compat (`lynx/core/renderer/template_assembler.cc:3231-3245`).
6. **Teardown** — when the owning `RadonLazyComponent` is removed from the tree, `PageProxy::EraseFromEmptyComponentMap`/`EraseComponent` (`lynx/core/renderer/page_proxy.cc:934-971`) detach it from the pending map; the entry itself (`TemplateEntry`, its VM, its `CSSStyleSheetManager`) is ref-counted off `TemplateEntryHolder::template_entries_` and dropped once nothing references it — there is no separate explicit "unload dynamic component bundle" API; it's ordinary node-removal + refcounting.

**Fallback / loading-state contract.** `RadonLazyComponent::AddFallback`/`RenderFallback` (`radon_lazy_component.cc:394-426`) implement a `<component is slot="fallback">`-style child slot: while `state_ == STATE_UNKNOWN` the fallback plug is just stored; it's only spliced into the tree (via a synthetic `RadonSlot`) if loading later fails (`STATE_FAIL`). **This is a fundamentally different loading-state model from ReactLynx's own `<component is>` implementation** (see below) — Radon's native fallback slot has no "in-flight/loading" render state (nothing shown while pending, only on hard failure), whereas ReactLynx wraps every dynamic component in `Suspense`, which does have a first-class loading state (the `Suspense` fallback shows immediately, not just on error).

**Reuse / diffing semantics.** `RadonLazyComponent::CanBeReusedBy` (`radon_lazy_component.cc:288-305`) requires matching `entry_name_` (the bundle URL) and matching `IsEmpty()` state for two lazy-component nodes across a diff to be treated as the same instance — i.e., changing the `is="..."` prop to a different bundle URL forces full teardown/rebuild, not an in-place prop update, matching how conditional/keyed component swapping works generally in Radon.

**CSS scope isolation (loading-lifecycle-adjacent, not just selector semantics).** Each lazy bundle's `TemplateEntry` owns its own `CSSStyleSheetManager` (`TemplateEntry::GetStyleSheetManager()`, `lynx/core/renderer/template_entry.h:102-104`), and `RadonLazyComponent::SetContext` re-points the freshly-loaded component's `style_sheet_manager_`/`intrinsic_style_sheet_` to that entry's manager, keyed by the mould's own `css_id()` (`radon_lazy_component.cc:158-165`) — so a lazy bundle's stylesheet is a wholly separate cascade scope from the parent page's, joined only by DOM containment, not partial merge. This is on top of (not the same finding as) the selector-visibility rules a separate CSS section presumably already covers.

**ReactLynx-level implementation is materially different from the native engine's.** Reading `lynx-stack/packages/react/runtime/src/snapshot/compat/componentIs.ts` shows ReactLynx does **not** rely on `RadonLazyComponent` directly for its own `<component is="x">` JSX sugar; instead it's a polyfill built entirely out of standard React primitives:
```
const D = useMemo(() => lazy(() => loadLazyBundle(is)), [is]);
return createElement(Suspense, { key: is }, createElement(D, props));
```
(`lynx-stack/packages/react/runtime/src/snapshot/compat/componentIs.ts:15-27`). `loadLazyBundle` (`lynx-stack/packages/react/runtime/src/core/lynx/lazy-bundle.ts:64-136`) branches on `__LEPUS__` vs `__JS__`: on the Lepus/main thread it calls the native builtin `__QueryComponent(source)` synchronously and wraps the result in a promise whose `.then` is monkey-patched to resolve **synchronously** (`makeSyncThen`, lines 11-56) so Suspense can render on first screen without an extra async tick; on the background/JS thread it calls `lynx.QueryComponent(source, callback)` (or `lynx.getNativeLynx().QueryComponent`) and, on success, looks the loaded module up via `lynxCoreInject.tt.getDynamicComponentExports(schema)`. Compile-time support is a dedicated SWC pass, `swc_plugin_dynamic_import` (`lynx-stack/packages/react/transform/crates/swc_plugin_dynamic_import/lib.rs`), which rewrites `import(url, {with:{type:"component"}})` into `loadLazyBundle(url)` calls and injects `lynx.loadLazyBundle` wiring; and a `jsx.isDynamicComponent` compile flag (`lynx-stack/packages/react/transform/swc-plugin-reactlynx/src/lib.rs`) that changes CSS-scope-id generation for the split-off entry.

**Critical gap confirmed by direct search: `__QueryComponent` and `lynxCoreInject.tt.getDynamicComponentExports` have zero implementation anywhere in `lynx-stack`.** Both are referenced only as externally-injected globals in tests (mocked via `vi.fn()`, e.g. `lynx-stack/packages/react/runtime/__test__/element-template/lynx/lazy-bundle.test.ts`); there is no TS or Rust body backing either symbol in `packages/web-platform/web-core*` or elsewhere in the monorepo — the web-platform target never actually resolves a lazy bundle at runtime, it is a pure host/native binding contract. This matches the task's hypothesis. **lynx-vello therefore has no existing reference implementation to port for the runtime side of dynamic component loading** — it must build the whole subsystem (resource fetch hook, per-entry decode via the already-built decoder, per-entry style/VM scoping, Suspense-compatible promise resolution, and success/fail event dispatch) from the native-engine spec (`RadonLazyComponent` + `LazyBundleLoader` + `TemplateEntry`) described above, while presenting it to ReactLynx-authored code through the same `lynx.QueryComponent`/`getDynamicComponentExports`/`__SetCSSId(..., entryName)` surface ReactLynx's compiled output already expects.

**Fiber-arch note.** The current engine has two rendering architectures; everything above is the legacy Radon-tree path. The newer Fiber-element tree has an analogous but distinct primitive, `FrameElement` (`lynx/core/renderer/dom/fiber/frame_element.h/.cc`), which loads a `LynxTemplateBundle` for an embedded `<frame src="...">` (cross-instance embedding, its own `FrameElementData`, `bindload` event) — this is architecturally close to lazy components (same `LazyBundleLoader`, same `LynxResourceType::kLazyBundle`/`kFrame` fetch path) but is a **different element type with different semantics** (whole nested Lynx page/instance vs. spliced-in component subtree) and was not the primary subject of this section; flagging it for a follow-up so it isn't conflated with `<component is>`.

### `<frame>` — nested Lynx-bundle embedding element

`frame` is a dedicated built-in element (Fiber architecture: `FrameElement : FiberElement`) whose sole job is to mount an **entire second, independent Lynx render root** — its own template bundle, its own element/DOM tree, its own CSS scope, and (per platform) its own JS/MTS runtime instance — as a rectangular leaf inside the host page's tree. This is architecturally distinct from both other "load more stuff" mechanisms in Lynx:

- **`<component is=…>` / `RadonLazyComponent`** (dynamic/lazy component, Radon architecture `RadonComponent` subclass, `lynx/core/renderer/dom/vdom/radon/radon_lazy_component.h`) fetches a bundle via the *same* `LazyBundleLoader`/resource-loader machinery, but **splices the fetched component's element tree into the existing page's tree** — same runtime, same CSS/JS scope, just asynchronously populated. It is not isolated.
- **`<web-view>`** (`lynx/platform/darwin/ios/lynx_xelement/webview/`, `lynx/platform/android/lynx_xelement/lynx_xelement_webview/`) wraps the platform's native web-view control (`WKWebView`/`android.webkit.WebView`) to load arbitrary **HTML/remote-web** content — nothing to do with Lynx template bundles.
- **`<frame>`** loads another **`.lynx.bundle`/`.web.bundle`** (a full Lynx template, referenced by `src` URL, resolved through `LazyBundleLoader::LoadFrameBundle` → `LynxResourceType::kFrame`, `lynx/core/resource/lazy_bundle/lazy_bundle_loader.cc:100`) and boots a **second complete render pipeline** for it, nested as a subtree of the host page.

**Native (iOS/Android) implementation.** `FrameElement` (`lynx/core/renderer/dom/fiber/frame_element.h/.cc`) is a `FiberElement` that: refuses children (`OnNodeAdded` logs `"frame element cannot adopt any child"` — it is a leaf/replaced-element, not a container that composes with host DOM children); on `src` attribute change, calls `element_manager_delegate()->LoadFrameBundle(src, this)`, deduplicated by src (`ElementManagerDelegateImpl::LoadFrameBundle`, `lynx/core/renderer/element_manager_delegate_impl.cc:26`) which in turn hits `LazyBundleLoader::LoadFrameBundle` (checks a pre-registered/cached bundle map first, else issues a real resource fetch tagged `kFrame`); once loaded, hands the decoded `LynxTemplateBundle` down to the platform paint layer via `element_container()->SetFrameAppBundle(bundle)` (`lynx/core/renderer/dom/base_element_container.cc:62` → `PaintingContextAndroid::SetFrameAppBundle`, `lynx/core/renderer/ui_wrapper/painting/android/painting_context_android.cc:877`). On Android the platform UI (`UIFrame`/`LynxFrameView`, `lynx/platform/android/lynx_android/.../ui/frame/LynxFrameView.java:89-172`) responds to the bundle by constructing a **brand-new `LynxTemplateRender`** (`mRender = new LynxTemplateRender(...)`) and calling `mRender.loadTemplate(...)` — i.e. it stands up a second, fully independent TASM/element-tree/layout/paint pipeline rooted at the `frame`'s `UIBodyView`, with its own JS engine instance. Layout-wise, `FrameShadowNode` (`lynx/platform/android/.../ui/frame/FrameShadowNode.java`) is a `CustomLayoutShadowNode`/`CustomMeasureFunc` — from the host layout tree's perspective the frame behaves like a **replaced element** (à la `<iframe>`/`<img>`): it reports an intrinsic content size (`updateIntrinsicContentSize`, driven by `auto-width`/`auto-height`/`preset-width`/`preset-height` props) but its internal content is laid out by the nested engine, opaque to the host's layout algorithm.

**Lifecycle events.** `frame` fires `bindload` (native custom event `"load"`, detail `{url, statusCode, statusMessage}`, tested in `lynx/core/renderer/dom/fiber/frame_element_unittest.cc`) plus a global Lepus event `frameLoaded` (`kFrameLoaded`, carries `{detail, frameElement}` so app code can inspect which frame fired) regardless of whether `bindload` is bound, and independently of load success/failure (error path still fires `frameLoaded`, confirmed by `FrameLoadFailureWithoutBindloadStillTriggersFrameLoaded` test). If the bundle finishes loading *before* the frame has a painting node yet (i.e., before first flush), the event is deferred and fired exactly once on the next `FlushProps` (`FrameDeferredLoadTriggersEventsOnceAfterFlush` test) — load/attach ordering is explicitly handled, not racy. A separate `bindloadmetrics` event (native/iOS/Android only, `since 3.9`) reports the nested page's own `LoadBundleEntry` performance-observer entry plus `mode` (`"embedded"`/`"standard"`, from `EmbeddedMode`).

**Public prop/event surface** (`lynx/js_libraries/types/types/common/element/frame.d.ts`, `frame.test-d.tsx`): `src` (required, string, the bundle URL — this is the *only* required prop), `data` (arbitrary object passed as init data to the nested page), `global-props` (passed to the nested page's `lynx.__globalProps`), `bindload`, `bindloadmetrics`, `auto-width`/`auto-height` (booleans, default false — let the frame's measured size follow the nested page's reported intrinsic content size instead of the host's constraint), `preset-width`/`preset-height` (px/rpx, size to use before the nested page reports an intrinsic size), `enable-multi-async-thread` (override whether the nested page uses the multi-threaded engine mode; inherits host setting if omitted). Notably, `embedded-mode` exists in the native/Java API (`EmbeddedMode.java`, bitflags `ENGINE_POOL`/`LAYOUT_IN_ELEMENT`/`FRAGMENT_LAYER_RENDER`/`USE_TEXT_SERVICE` for perf-tuning many-frames-per-page scenarios, e.g. feed-like UIs) but is **explicitly excluded** from the public TS surface (`HasEmbeddedMode` test asserts `false`) — it's an internal/experimental knob, not part of the stable contract lynx-vello needs to match.

**Web target (`web-core`) implementation — already real, not a gap.** Contrary to the initial hypothesis that this might be unimplemented on web, `web-core` fully implements `frame`: the tag-to-DOM map (`lynx-stack/packages/web-platform/web-core/ts/constants.ts:90`) maps Lynx tag `'frame'` → HTML tag **`lynx-view`** — i.e., a nested `frame` is realized as **another `<lynx-view>` custom element mounted inside the outer `<lynx-view>`'s tree** (`__CreateFrame` in `lynx-stack/packages/web-platform/web-core/ts/client/mainthread/elementAPIs/createElementAPI.ts:195`, dispatch wired in `lynx-stack/packages/react/runtime/src/snapshot/snapshot/definition.ts:190`). `LynxViewElement` (`lynx-stack/packages/web-platform/web-core/ts/client/mainthread/LynxView.ts`) is the exact same top-level element app authors use to embed a whole Lynx app into a plain web page — so `frame` is literally "recursively embed another `lynx-view`". Each `LynxViewElement` instance uses **Shadow DOM** (`attachShadow({mode:'open'})`, own `adoptedStyleSheets`) for style/DOM encapsulation, and creates an isolated JS execution context per instance via `createIFrameRealm()` (`lynx-stack/packages/web-platform/web-core/ts/client/mainthread/createIFrameRealm.ts`) — a real (display:none, sandboxed) `<iframe>` used purely as a JS-global sandbox for main-thread-script (MTS) execution, not as the frame's visual/layout container. `dispatchLynxViewLoadEvent` fires a `load` CustomEvent with `{statusCode, statusMessage, url}` matching native's `bindload` detail shape.

**Consequence for lynx-vello.** "Multiple isolated rendering scopes in one page" is real and must be a first-class primitive: implementing `frame` means lynx-vello needs the ability to instantiate a second, fully independent decode→style→layout→paint pipeline (its own stylo `Stylist`/rule tree, its own layout tree, its own element ID/event-target namespace, its own JS-equivalent runtime state) rooted at an arbitrary point in the host's render tree, composited as a black-box replaced element from the host layout's perspective (host asks it for an intrinsic/measured size; never reaches into its internals). This is **not** iframe-style full process/security isolation (same address space, same GPU surface/vello scene in both native and — architecturally, if lynx-vello were to run in-process rather than in-browser — this repo's target); it *is* full CSS-cascade/DOM-tree/JS-global isolation. No W3C standard directly corresponds to this element (closest analogues are `<iframe>` for isolation-as-replaced-element and `<embed>`/`<object>` for embedding another document by URL); Lynx's `frame` is a bespoke primitive, not a standards deviation to flag — the compliance question is really "closest W3C analogy" rather than "correct vs incorrect."
