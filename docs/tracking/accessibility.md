# Accessibility (a11y) node tree & APIs

> Research: multi-agent sweep over `lynx/` and `lynx-stack/` (see [AGENTS.md](../../AGENTS.md) for the reference-repo shorthand and the W3C-first standards policy). Supersedes the earlier stub.

### Accessibility (a11y) node tree, screen-reader semantics & JS Accessibility API

Native Lynx has a substantial, platform-specific accessibility subsystem layered on top of the regular UI tree — it is **not** derived from the DOM/CSS box tree the way ARIA-in-HTML is; it is a parallel "virtual accessibility tree" that each platform's LynxUIOwner/LynxBaseUI walks and projects onto the native a11y API (Android `AccessibilityNodeInfo`/`AccessibilityNodeProvider`, iOS `UIAccessibility`, HarmonyOS `@ohos.accessibility`). Android alone ships **three different, mutually-exclusive implementation strategies** selected by page config, which matters for lynx-vello because it means "port the Android accessibility code" is not a single well-defined target:

- **Legacy `LynxAccessibilityNodeProvider`** (`lynx/platform/android/lynx_android/src/main/java/com/lynx/tasm/behavior/ui/LynxAccessibilityNodeProvider.java`) — default when neither `enableNewAccessibility` nor `enableA11y` page-config flags are set. Extends Android's `AccessibilityNodeProvider` directly; builds a flat virtual-child list by walking the flattened UI tree (`findAllAccessibilityNodeOfLynxDeep`), sorts by screen position (top→bottom in `SECTION_COUNT=50`-row bands, then left→right), and answers `createAccessibilityNodeInfo`/`performAction`/`onHover`.
- **New `LynxAccessibilityDelegate` + `LynxNodeProvider`** (`lynx/platform/android/.../accessibility/LynxAccessibilityDelegate.java`) — used when `enableNewAccessibility` is set. Extends `AccessibilityDelegateCompat`, adds explicit accessibility-focus state tracking (`mAccessibilityFocusedVirtualViewId`), hover dispatch via Lynx's own `hitTest`, and fires synthetic `tap`/`click` touch events into the JS event system when `ACTION_CLICK` is invoked on a virtual node with `accessibility-enable-tap`.
- **`LynxAccessibilityHelper`** (`lynx/platform/android/.../accessibility/LynxAccessibilityHelper.java`) — used when `enableA11y` is set ("a11y mode"); forces real (non-flattened) Android Views for every accessible node instead of a virtual tree, and defines the `accessibility-traits` enum (`NONE`/`IMAGE`/`BUTTON`, mapped to fake View class names `android.widget.ImageView`/`android.widget.Button` so TalkBack announces a role) plus the "exclusive focus" subtree map.

All three are orchestrated by `LynxAccessibilityWrapper` (`lynx/platform/android/.../accessibility/LynxAccessibilityWrapper.java`), which also owns: system a11y/touch-exploration state monitoring (`LynxAccessibilityStateHelper`), the mutation-observer machinery (`LynxAccessibilityMutationHelper`, action enum `INSERT/REMOVE/DETACH/UPDATE/STYLE_UPDATE`, flushed to JS as global event `"a11y-mutations"`), and cloud-tag/JS-facing methods (`requestAccessibilityFocus`, `fetchAccessibilityTargets`, `innerText`) that are exposed per-UI as `@LynxUIMethod`s (`element.invoke('requestAccessibilityFocus', ...)` from JS) as well as through the native module described below. Focusing an element also fires global event `"activeElement"` with `{element-id, a11y-id}` — this is the mechanism ReactLynx apps hook to know what the screen reader focused.

On iOS, `LynxUI+Accessibility.m` and `LynxUIOwner+Accessibility.m` implement the same conceptual features on top of `UIAccessibility`/`UIView.accessibilityElements`: custom element ordering via `accessibility-elements`/`accessibility-elements-a11y` (supports mixing element signs and `a11y-id` string selectors, re-sorted by DOM y/x position when ambiguous), `accessibility-exclusive-focus` (walks up the tree marking every sibling subtree `accessibilityElementsHidden = YES`), and a **W3C-relevant quirk**: iOS `UIAccessibility` has no native concept of nested accessibility elements (a container marked `isAccessibilityElement` cannot itself contain focusable descendants), so Lynx works around it in `checkNestedAccessibilityElements` by synthesizing an invisible dummy `UIView` sibling to carry the parent's label/traits when children need to be independently focusable. HarmonyOS's `@ohos.accessibility`-based path was not enumerated file-by-file here beyond the JS module (see below); it was not the focus of the read but is referenced as a third platform target in `platform/harmony/lynx_harmony/src/main/ets/`.

The CSS-adjacent props are parsed as ordinary component attributes, not CSS properties: the base allow-list in `lynx/core/renderer/dom/component_attributes.cc` (`ComponentAttributes::GetAttrNames`) includes only `accessibility-label`, `accessibility-element`, `accessibility-traits` (plus unrelated `focusable`/`focus-index`) — the fuller prop surface (`accessibility-elements`, `accessibility-elements-a11y`, `accessibility-enable-tap`, `accessibility-exclusive-focus`, `accessibility-value`, `accessibility-heading`, `accessibility-status`, `accessibility-actions`, `accessibility-role-description`, `a11y-id`, `android-accessibility-keep-focused`) is declared per-platform in `PropsConstants.java` (Android, lines 116-134) and wired via `@LynxProp`-annotated setters on `LynxBaseUI.java` (lines 2061-2356) / `LYNX_PROP_DECLARE`/`LYNX_PROP_DEFINE` macros in `LynxUI.m` (iOS, lines 1669-4610). There is **no `accessibility-focusable` prop** in either codebase searched — ordinary `focusable`/`focus-index` (tab/D-pad focus order, a separate non-a11y-specific mechanism) is what the task hint's name likely refers to; it was not found as an a11y-specific attribute name.

The JS-facing `LynxAccessibilityModule` native module is a genuinely tiny, uniform surface across all three platforms — confirmed by reading Android (`lynx/platform/android/lynx_android/src/main/java/com/lynx/jsbridge/LynxAccessibilityModule.java`), iOS (`lynx/platform/darwin/ios/lynx/module/LynxAccessibilityModule.mm`), and HarmonyOS (`lynx/platform/harmony/lynx_harmony/src/main/ets/jsbridge/LynxAccessibilityModule.ets`): only two methods, `registerMutationStyle({mutation_styles: string[]}, callback)` (registers which CSS style keys should generate `"a11y-mutations"` events when they change) and `accessibilityAnnounce({content: string}, callback)` (Android: `LynxView.setAccessibilityPaneTitle`/`announceForAccessibility`; iOS: `UIAccessibilityPostNotification(UIAccessibilityAnnouncementNotification, ...)`; Harmony: `accessibility.sendAccessibilityEvent` with `type: 'announceForAccessibility'` — this is the ARIA-live-region-"polite announcement" equivalent). Per-element methods (`requestAccessibilityFocus`, `fetchAccessibilityTargets`, `innerText`) are exposed as UI methods (invoked via element refs), not through this NativeModule.

**Web-target gap, confirmed by direct grep**: `packages/web-elements/src` and `packages/web-platform/web-core/src` (and the whole `packages/web-platform` tree) contain zero source-level hits for `accessibility`, `accessibility-element`, `accessibility-label`, `accessibility-traits`, `a11y`, etc. The only repo hits are inside compiled build artifacts (`web-core/dist/.../lynx-core-chunk.js`, e2e/test fixture JSON) that embed the transpiled `js_libraries/lynx-core` runtime, which merely binds `this.accessibilityAnnounce = this.getNativeApp().nativeModuleProxy.LynxAccessibilityModule...` generically like any other NativeModules proxy call — inspected directly and confirmed this is a dead reference on web (no native module ever registers `LynxAccessibilityModule` for the web target), not an implementation. Documented evidence that the config knob exists in the build tooling regardless of runtime support: `enableAccessibilityElement` in `@lynx-js/react-rsbuild-plugin` (`lynx-stack/website/docs/en/api/react-rsbuild-plugin.pluginreactlynxoptions.enableaccessibilityelement.md`, default `false`) — this option controls the compiled bundle's default `accessibility-element` value for `<view/>`, but web-core's runtime never consumes it. **This means lynx-vello has literally no prior art to port from this axis and must design the a11y tree/props/module from the native (Android/iOS) behavior described above, targeting whatever native a11y API the eventual OS target(s) expose** (e.g., AccessKit via winit/muda-style crates, or platform APIs directly) — this is a from-scratch subsystem, not a port.
