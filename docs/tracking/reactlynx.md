# ReactLynx runtime, compiler model & public API

> Research: multi-agent sweep over `lynx/` and `lynx-stack/` (see [AGENTS.md](../../AGENTS.md) for the reference-repo shorthand and the W3C-first standards policy). Supersedes the earlier stub.

### Architecture: from JSX to a running dual-thread app

**Compiler (Rust/SWC, `lynx-stack/packages/react/transform`).** ReactLynx does not ship a runtime `jsx()`/`jsxs()` factory that builds a vdom tree of native-element descriptors at call time (unlike react-dom). Instead, `@lynx-js/react/transform` is an SWC-plugin pipeline (`lynx-stack/packages/react/transform/src/lib.rs`) that statically compiles each JSX subtree at build time into a **snapshot**: a pair of a `create(ctx)` function (imperatively builds native `FiberElement`s once, via globals like `__CreateView`/`__CreateText`/`__CreateImage`/`__SetAttribute`) and a `update[i](ctx, index, oldValue)` array (one updater per dynamic slot). The pipeline order (as wired in `transform/src/lib.rs`, `~L380-720`) is roughly: `simplify` → `directive_dce` (strips `"background only"`/`"use lepus only"` blocks per target, `transform/crates/swc_plugin_directive_dce/lib.rs:63-84`) → `define_dce` → `css_scope` (`swc_plugin_css_scope`) → React JSX normalization (`swc_core::ecma::transforms::react`) → **`SnapshotJSXTransformer`** (`transform/crates/swc_plugin_snapshot/lib.rs`, 3357 lines; entry at `visit_mut_jsx_element`, `lib.rs:520`) *or*, if the newer opt-in "element template" backend is enabled, `ElementTemplateTransformer` (mutually exclusive with snapshot, `transform/src/lib.rs:505-506`) → `swc_plugin_list` (list/list-item static analysis) → `swc_plugin_text` (SDK ≥3.1 text optimization) → `shake` (dead snapshot elimination) → `compat`/`compat_post` (React-17-shape compat) → `refresh` (Fast Refresh) → **`swc_plugin_worklet`** (compiles `"main thread"`/`"main-thread"` and `"use worklet"` function bodies into serializable worklets, `transform/crates/swc_plugin_worklet/worklet_type.rs:8-16`) → `dynamic_import` → `inject`. Attribute-name classification (`transform/crates/swc_plugin_snapshot/attr_name.rs:25-89`) recognizes: plain attrs, `data-*` → dataset, `class`/`className`, `style`, `id`, `ref`, `__lynx_timing_flag`, and event props matched against `^(global-bind|bind|catch|capture-bind|capture-catch)([A-Za-z]+)$` (Lynx's four/five event dispatch phases — non-standard vs. DOM's capture/bubble two-phase model). A namespaced attr `xxx:bindtap={fn}` (`attr_name.rs:61-75`) becomes a `WorkletEvent(worklet_type, event_type, event_name)` — this is the compiler-level home of the **`main-thread:` event-handler directive**: `main-thread:bindtap={handler}` statically marks `handler`'s body to run as a Main Thread Script (MTS) worklet rather than a background-thread JS closure.

Static (compile-time-constant) parts of JSX become part of `create`; anything referencing a variable/expression becomes a numbered **dynamic slot** recorded in the snapshot's `slot: [DynamicPartType, number][]` array (types: `Slot`, `MultiChildren`, `Children`, `ListChildren`, `SlotV2`, `ListSlotV2` — `runtime/src/snapshot/snapshot/dynamicPartType.ts`, referenced from `runtime/src/snapshot/snapshot/definition.ts:42-52`). At runtime each JSX invocation of a compiled component instantiates a `SnapshotInstance` (`runtime/src/snapshot/snapshot/snapshot.ts:139-613`) — a lightweight tree node compatible with Preact's `ContainerNode` interface (`parentNode`/`childNodes`/`insertBefore`/`removeChild`), so **Preact's real reconciler drives the tree** on the background thread; `SnapshotInstance` is simply the "DOM" Preact diffs against. `ensureElements()` (`snapshot.ts:181-276`) lazily realizes native elements from a snapshot instance by walking its `slot` plan; `setAttribute`/`callUpdateIfNotDirectOrDeepEqual` (`snapshot.ts:544-612`) invoke the compiled per-slot updater only when the new value isn't reference-or-deep-equal to the old (`isDirectOrDeepEqual`), which is the mechanism giving ReactLynx cheap "don't touch the native tree if the prop didn't really change" behavior without a general vdom diff of DOM attributes.

**Dual-thread split and Instant First-Frame Rendering (IFR).** The same compiled JSX/hook code ships to *both* threads, gated by build-time globals `__MAIN_THREAD__`/`__BACKGROUND__`/`__LEPUS__`/`__JS__`, but each thread installs a different hook/runtime implementation:
- **Main thread ("Lepus")**: `runtime/src/snapshot/lifecycle/render.ts:14-36` (`renderMainThread`) does a *single, one-shot* Preact `render(__root.__jsx, ...)` to materialize first-screen elements (`renderToOpcodes/index.ts`) as fast as possible — "Instant First-Frame Rendering". To make this safe with no background thread alive yet, `runtime/src/core/hooks/mainThreadImpl.ts` (`installMainThreadHooks`, `L49-224`) patches Preact's `options` hook table with a **parallel hooks implementation**: `useState`/`useReducer`'s setter is a no-op that only `console.error`s in dev (`mainThreadImpl.ts:106-131`, "Cannot update state in main thread!"); `useEffect`, `useLayoutEffect`, and `useImperativeHandle` are `noop` (`mainThreadImpl.ts:206-208`); `useMemo` always recomputes (ignores the deps array, `mainThreadImpl.ts:143-147`, comment: "used for first screen and need not to cache value by args"); `useContext`, `useDebugValue`, `useErrorBoundary` (no-op boundary, doesn't actually recover), and `useId` are reimplemented from scratch mirroring Preact's own hook internals (this file's header literally says "modified from preact/hooks"). This main-thread render path is a one-time SSR-like paint, not a live component tree — it is thrown away once hydration completes.
- **Background thread ("JS")**: uses real `preact/hooks` (`runtime/src/core/hooks/react.ts`) with working state, effects, refs, context, and error boundaries, wrapped only with optional perf-profiling instrumentation (`buildTraceOption`/`profileStart`/`profileEnd`, `react.ts:32-123`). This is where the actual, persistent component tree lives and where all real interactivity/state updates happen.
- **Hydration** (`runtime/src/snapshot/renderToOpcodes/hydrate.ts`, `hydrate()` at `L223-431`) reconciles the background thread's freshly-rendered `SnapshotInstance` tree ("after") against the main thread's already-painted tree ("before"): it transfers `__elements`/`__element_root` ownership from before→after, replays only-changed `setAttribute`s (`L236-244`), and recursively hydrates children per slot type — for `Children`/`SlotV2` slots via a **keyed diff** (`diffArrayLepus`, `workletEvent.ts` — wait, actually defined alongside in `renderToOpcodes/hydrate.ts:37-100`, keyed by `type` for plain children or by `item-key` for list children, `L55-65`) that computes insert/remove/move sets exactly like React's list-reconciliation algorithm, then applies them via `diffArrayAction` (`L160-216`) which calls into native `__InsertElementBefore`/`__RemoveElement`. For `list`/`list-item` slots specifically, hydration additionally rewires the native list's recycling pool (`gSignMap`/`gRecycleMap`, populated by `componentAtIndexFactory`/`enqueueComponentFactory` in `runtime/src/snapshot/list/list.ts:9-320`) and calls `__UpdateListCallbacks`/`__SetAttribute(..., 'update-list-info', info)` (`hydrate.ts:389-416`) to tell the native list host which cells were inserted/removed/moved.
- **Steady-state updates**: `runtime/src/snapshot/lifecycle/patch/commit.ts` (`replaceCommitHook`, `L94-176`) hijacks Preact's `options[COMMIT]` hook on the background thread. After each Preact commit it drains a **global, minimal, serializable patch buffer** (`__globalSnapshotPatch`, `runtime/src/snapshot/lifecycle/patch/snapshotPatch.ts`) whose opcodes are exactly: `CreateElement(type, id)`, `InsertBefore(parentId, childId, beforeId, slotIndex?)`, `RemoveChild(parentId, childId)`, `SetAttribute(id, dynamicPartIndex, value)`, `SetAttributes(id, values)`, plus `nodesRefInsertBefore/RemoveChild` for portals and `DEV_ONLY_*` ops for HMR (`snapshotPatch.ts:11-75`). This patch list is JSON-serialized (`commitPatchUpdate`, `commit.ts:181-215`) and sent to the main thread over `lynx.getNativeApp().callLepusMethod(LifecycleConstant.patchUpdate, obj, callback)` (`commit.ts:164-171`), where it's replayed against the live main-thread `SnapshotInstance` tree (`snapshotPatchApply.ts`, not detailed above but referenced by name in `commit.ts:31`) to mutate native elements. This is the entire cross-thread update contract — background does full "real" React/Preact reconciliation privately; only these opcodes cross the thread boundary.
- An **experimental alternative backend**, "element template" (`__USE_ELEMENT_TEMPLATE__`, code under `runtime/src/element-template/`, compiled by `ElementTemplateTransformer` instead of `SnapshotJSXTransformer` when enabled), exists in this same package with its own hydration/patch/list/worklet stack (e.g. `element-template/background/hydrate.ts`, `ElementTemplateUpdateOps`). It mirrors the snapshot architecture closely (same IFR/hydrate/patch shape) but is a distinct, newer code path; lynx-vello should treat classic **snapshot** as the primary target and element-template as a secondary/future variant, not conflate the two when reading source.

**Main Thread Scripting (MTS) / `main-thread:` directive.** Two distinct worklet kinds are compiled by `swc_plugin_worklet` (`transform/crates/swc_plugin_worklet/worklet_type.rs:8-16`): directive `"main thread"`/`"main-thread"` → `WorkletType::Element` (type string `"main-thread"`) for element-scoped handlers (event bindings, `main-thread:ref`, gestures — see JSX test fixtures at `transform/crates/swc_plugin_snapshot/lib.rs:2853-2907`: `main-thread:bindtap`, `main-thread:ref`, `main-thread:gesture`); directive `"use worklet"` → `WorkletType::UI` (type `"ui"`) for UI-thread animation worklets. At the runtime layer, `updateWorkletEvent` (`runtime/src/snapshot/snapshot/workletEvent.ts:35-71`) special-cases `workletType === 'main-thread'`: it calls `retainWorkletCtx`/`onWorkletCtxUpdate` and registers the worklet via `__AddEvent(element, eventType, eventName, {type:'worklet', value})`, so the native engine invokes the worklet directly on the main thread without round-tripping to background JS — this is what gives MTS handlers (e.g. `main-thread:bindtap`) synchronous, no-message-passing UI responsiveness. Background→main-thread **imperative** calls use a separate mechanism, `runOnMainThread()` (`runtime/src/core/thread-function-call/main-thread.ts:33-62`): wraps a function as a worklet, dispatches a `runWorkletCtx` event over `lynx.getCoreContext()`, and resolves a `Promise` when the main thread replies (`onFunctionCall`/`return-value.ts`), gated behind SDK version checks (`isMainThreadFunctionSupported` requires SDK > 2.13, `main.ts:64-70`).

**Refs to native elements.** `ref`/`main-thread:ref` values are validated (`assertValidRef`, `runtime/src/core/ref.ts:27-38`) to be either a callback or a `{current}` object; when applied (`applyOrdinaryRef`, `L47-72`) they receive a **proxy object** wrapping `lynx.createSelectorQuery().select(selector)` (`SelectorRefProxy`, `L122-166`), not a raw native handle — method calls on the ref (e.g. `.setNativeProps()`-like NodesRef methods) are queued and only actually executed via `.exec()`, matching Lynx's async `SelectorQuery`/`NodesRef` API rather than a synchronous DOM-node reference.

**List reuse-identifier diffing.** `<list-item key item-key reuse-identifier full-span>` (fixtures at `transform/crates/swc_plugin_list/lib.rs:245,328-482`) drives two independent mechanisms: (1) Preact-level keyed reconciliation during hydration/update, keyed by `item-key` (`diffArrayLepus(..., isListHasItemKey=true)`, `renderToOpcodes/hydrate.ts:55-65`); (2) native cell-recycling, keyed by `type + reuse-identifier` (`runtime/src/snapshot/list/list.ts:115,313`), where `componentAtIndex`/`componentAtIndexes` (native → JS callback invoked when the platform's virtualized list needs a cell rendered) pull a `SnapshotInstance` either from a same-identity recycle pool (`gRecycleMap`) or freshly `ensureElements()`, and `enqueueComponent` (native → JS, cell scrolled off-screen) returns it to the pool for reuse (`list.ts:297-320`). This two-level design (React reconciliation identity via `item-key`, native view-recycling identity via `reuse-identifier`) has no single equivalent in vanilla React and must be modeled explicitly in lynx-vello's list widget.

**Context/Fragment/Suspense/Portal/error-boundary provenance.** `Component`, `createContext`, `Fragment`, `createRef`, `forwardRef`, `isValidElement`, `memo`, `lazy`, `useSyncExternalStore`, `Children` are all re-exported straight from `preact`/`preact/compat` (`runtime/src/index.ts:7-18,42-92`) — so class-component lifecycle (`componentDidMount/Update/WillUnmount`, `shouldComponentUpdate`, `getDerivedStateFromProps`, `componentDidCatch`, `getDerivedStateFromError`) works exactly as in Preact on the background thread with **zero ReactLynx-specific reimplementation**; only the hook layer (`core/hooks/react.ts`) and `main-thread` render path get ReactLynx-specific shims. `Suspense` (`runtime/src/snapshot/lynx/suspense.ts:14-58`) is a thin wrapper around `preact/compat`'s `Suspense` that swaps `createElement` implementations by thread (`createElementMainThread` from `@lynx-js/react/lepus` vs. background's `createElementBackground`) — this means Suspense *is* present on the main-thread IFR pass too (unlike effects/state). `createPortal` (`runtime/src/snapshot/lynx/portals.ts:163-179`) explicitly returns `null` when `__MAIN_THREAD__` is true (a portal only exists once hydrated on the background thread) and its "container" is a `NodesRef` (an already-selected native node reached via selector query), not an arbitrary tree node — background patches for portaled children are emitted as the dedicated `nodesRefInsertBefore`/`nodesRefRemoveChild` snapshot ops rather than ordinary parent/child ops.

**W3C-divergence flags relevant to this section** (per task instructions — Lynx-specific behavior lynx-vello should NOT copy where it diverges from web standards):
1. Lynx's event-binding grammar (`bind*`/`catch*`/`capture-bind*`/`capture-catch*`/`global-bind*`, `transform/crates/swc_plugin_snapshot/attr_name.rs:78`) is a 4-5-way dispatch-phase model, not the DOM's two-phase (capture/bubble) `addEventListener(type, fn, {capture})` model with `stopPropagation()`. This is inherently non-standard and out of scope for this section (belongs to the events/gestures tracking doc), but flagged here because it surfaces at the JSX-attribute level the compiler parses.
2. `useLayoutEffect` is explicitly documented as deprecated and is a **plain alias for `useEffect`** in ReactLynx (`runtime/src/core/hooks/react.ts:154-156`, doc comment: "cannot offer the precise timing for reading layout information and synchronously re-render, which is different from React"). W3C/React-correct behavior: `useLayoutEffect` should fire synchronously after DOM mutations but before the browser paints, blocking visual update until layout-reading effects complete. lynx-vello, aiming for React-standard behavior in a native compositor, should give `useLayoutEffect` real pre-paint synchronous timing (relative to vello's frame commit) rather than reusing this ReactLynx alias-to-`useEffect` shortcut, if true React compatibility is desired — but note the *goal statement's* ReactLynx-compatibility priority may mean matching the current (deprecated/aliased) behavior is actually what's wanted; flagging so the implementation decision is explicit rather than accidental.
3. Main-thread `useState`'s setter silently no-ops (only a dev console warning, `mainThreadImpl.ts:122-126`) rather than throwing or being statically disallowed — a deliberate ReactLynx compromise for the throwaway IFR pass, not a web standard to preserve; not itself a W3C divergence (no DOM/CSS equivalent) but worth noting as intentional non-standard React semantics unique to this render pass.

Not confirmed by source (flagging rather than guessing): the precise wire schema/versioning contract for `LifecycleConstant.patchUpdate` payloads beyond the `PatchList`/`Patch` shapes shown in `commit.ts:54-77`; exact behavior of `useSyncExternalStore` under the main-thread IFR hook shims (it is re-exported from `preact/compat` unmodified per `runtime/src/index.ts:17`, so it likely is not part of the `mainThreadImpl.ts` reimplemented hook set — this was not independently verified against `mainThreadImpl.ts`'s hook list, which does not mention it).

### ReactLynx Public API Surface: Components, Hooks & Utility APIs

Scope: everything a userland ReactLynx app actually `import`s from `@lynx-js/react` (and its subpath exports), `@lynx-js/react/runtime-components`, and the adjacent `@lynx-js/gesture-runtime` / `@lynx-js/motion` packages, plus the built-in JSX intrinsic elements (`<view>`, `<text>`, etc.) whose prop/event shape is defined in the vendored `@lynx-js/types` package (found at `lynx-stack/node_modules/.pnpm/@lynx-js+types@3.7.0/…`, consumed by `lynx-stack/packages/react/types/react.d.ts`). The package's `exports` map is `lynx-stack/packages/react/package.json`; the actual runtime source lives in `lynx-stack/packages/react/runtime/src/`. Everything under `@lynx-js/react/internal` is transform-generated glue (compiled output of JSX by `@lynx-js/react-transform`), not something userland code calls directly, so it is out of scope except where noted.

Event props follow the pattern `bind<name>` / `catch<name>` / `capture-bind<name>` / `capture-catch<name>` / `global-bind<name>` for every event (defined generically in `lynx-stack/node_modules/.pnpm/@lynx-js+types@3.7.0/node_modules/@lynx-js/types/types/common/events.d.ts`). This is a fundamentally different event-binding model from DOM's `addEventListener`/`on*` — confirmed as a real Lynx engine concept (not a web-platform invention) via `lynx/core/renderer/dom/element.cc` and `fiber_element.cc`. `catch` stops propagation (like `stopPropagation()`), `capture-*` fires during the capture phase, `global-bind` is Lynx-specific (no DOM equivalent — akin to a page-level event bus subscription). lynx-vello should implement bind/catch/capture semantics on top of a standard W3C capture+bubble event model internally, exposing the Lynx prop names as sugar.

### Notable gaps / things NOT found as public API (confirmed by search, not guessed)
- No `useEventDefer`, no standalone exported `useComponent`/`useLifecycle` hooks were found in `@lynx-js/react`'s public surface.
- Gesture builder API (`Gesture`, `useGesture`) lives in the separate `@lynx-js/gesture-runtime` package (`lynx-stack/packages/lynx/gesture-runtime`), not inside `packages/react` — flagging in case lynx-vello wants to bundle it as part of "ReactLynx compatibility" anyway, since real apps commonly pair the two.
- `@lynx-js/motion` is a third, fully optional adapter package, unrelated to `@lynx-js/react` exports; included for completeness since it's part of the "userland imports" surface but is low priority.
