# web-core dual-thread runtime architecture

> Research: multi-agent sweep over `lynx/` and `lynx-stack/` (see [AGENTS.md](../../AGENTS.md) for the reference-repo shorthand and the W3C-first standards policy). Supersedes the earlier stub.

### Scope and what changed vs. the "obvious" starting point

The task brief pointed at `lynx-stack/packages/web-platform/web-core` and expected a plain TS main-thread/worker split. The package has since been substantially rewritten: the hot-path DOM/CSS/event logic is now a Rust→WASM core (`web-core/src/...`), with a thin TypeScript "client runtime" (`web-core/ts/client/...`) gluing it to the browser. Its own `AGENTS.md` (`lynx-stack/packages/web-platform/web-core/AGENTS.md`) documents this split and is the best first read for anyone touching this package. `web-mainthread-apis` (a package name that sounds like it should hold this logic) is now dead — it has only a `tsconfig.tsbuildinfo`, no `src/`; all main-thread element APIs live in `web-core` today. The actual ReactLynx reconciler that drives the element-tree mutations from application code lives in a third package, `lynx-stack/packages/react/runtime/src` (snapshot-based VDOM, opcode generation, worklets) — out of scope for this section's file-level citations but referenced for context since it's the thing that *calls* the APIs described below.

### The two "threads" are not what they sound like

Lynx's native engine has a main thread (UI/layout/rendering, synchronous, has direct element-tree access) and a background thread (BTS: runs the application/React logic, async, message-passing only). web-core's browser mapping is:

- **Main Thread Script (MTS) realm = a sandboxed same-origin `<iframe>`**, not the real top-level main thread and not a Worker. Created by `createIFrameRealm` (`lynx-stack/packages/web-platform/web-core/ts/client/mainthread/createIFrameRealm.ts`): a hidden `<iframe sandbox="allow-same-origin allow-scripts">` is appended into the `<lynx-view>`'s shadow root, its `srcdoc` posts a `"lynx:mtsready"` message back to the parent once its own document is alive, and the returned `JSRealm` exposes `globalWindow` (the iframe's `contentWindow`), `loadScript` (async, via a `<script src>` tag + `module.exports` convention) and `loadScriptSync` (synchronous, via a blocking `XMLHttpRequest` + inline `<script>` — used because Lepus/mts historically expects synchronous script evaluation). Rationale: MTS code needs synchronous, in-process access to the real DOM (to create/mutate elements at "native UI operation" latency) and its own isolated global scope (so it doesn't pollute/get polluted by the host page's `window`), but it is *not* meant to run arbitrary untrusted JS the way BTS's Worker sandbox is — hence "iframe, not worker."
- **Background Thread Script (BTS) = a real dedicated `Worker`.** Bootstrapped by `BackgroundThread.startWebWorker` (`lynx-stack/packages/web-platform/web-core/ts/client/mainthread/Background.ts`), which lazily creates a `new Worker(new URL('../background/index.js', ...), { type: 'module', name: 'lynx-bg' })` (or shares one keyed by `lynxGroupId` across multiple `<lynx-view>` cards — see `BackgroundThread.contextIdToBackgroundWorker`), then hands it one port of a `MessageChannel` in the worker-start message. The worker entry (`lynx-stack/packages/web-platform/web-core/ts/client/background/index.ts`) reads `SystemInfo`/init payload off the initial `onmessage` and calls `startBackgroundThread` (`.../background/background-apis/startBackgroundThread.ts`), which wraps the port in an `Rpc`, builds a `nativeApp` (`createNativeApp.ts`) and the global `lynx` object (`createBackgroundLynx.ts`), dynamically imports `@lynx-js/lynx-core/web` (the shared, target-agnostic React/Lepus bootstrap — vendored from `lynx/js_libraries/lynx-core` — that exports `loadCard`/`destroyCard`/`nativeGlobal`), and finally registers a handler so the *main* side's `BackgroundThreadStartEndpoint` RPC call triggers `loadCard(nativeApp, {initData, cardType, updateData: initData}, nativeLynx)`. This is where ReactLynx's actual render tree gets constructed inside the worker.
- Both realms are owned/orchestrated by a single `LynxViewInstance` (`lynx-stack/packages/web-platform/web-core/ts/client/mainthread/LynxViewInstance.ts`), instantiated per `<lynx-view>` element (`LynxView.ts`'s `LynxViewElement.#render()`). It holds `mainThreadGlobalThis` (the iframe's globalThis), `mtsRealm` (the `JSRealm` handle), and `backgroundThread` (the `BackgroundThread` wrapper around the worker + its `Rpc`).

### The RPC transport (this is the thing lynx-vello has to behaviorally replace)

`lynx-stack/packages/web-platform/web-worker-rpc/src/Rpc.ts` implements a `MessagePort`-based RPC layer used for *every* main↔background call. Two call shapes exist, both important to replicate the semantics of, not just the plumbing:

- **Async (`isSync:false`)**: `postMessage({name, data, sync:false, retId?})`; if the endpoint `hasReturn`, the receiver's reply is routed back through a synthetic one-shot "retId" endpoint and resolves a `Promise`. Handlers can be registered *after* a message arrives — `Rpc` caches unhandled messages by name (`#messageCache`) and replays them once `registerHandler` is finally called, and `Rpc.postMessage` itself queues outgoing messages until `setMessagePort` is called (both matter because MTS/BTS boot asynchronously and out of lockstep with each other — many endpoints are registered lazily via `registerHandlerLazy`, e.g. `registerUpdateDataHandler.ts` waits for `tt.updateCardData` to exist on the `NativeTTObject` before wiring the handler, buffering any `updateData` calls that raced ahead).
- **Sync (`isSync:true`)**: allocates a `SharedArrayBuffer`-backed `Int32Array` lock plus an optional result buffer, posts the message, then **blocks the calling thread** with `Atomics.wait(lockViewer, 0, 0)` until the other side's handler runs and calls `Atomics.store`/`Atomics.notify`. The result (JSON-stringified into the shared buffer) is decoded synchronously on wake. This is a *real* synchronous cross-thread call — it requires cross-origin-isolation (COOP/COEP) for `SharedArrayBuffer`, and it is how e.g. `__invokeUIMethod`/`__setNativeProps`/`__getPathInfo` get a return value back to BTS without the caller needing to `await`. **This is the single hardest primitive for lynx-vello to replicate behaviorally**: a native engine without two OS threads separated by a serialization boundary can trivially make these calls synchronous (they're just direct function calls in-process), but must still preserve the *ordering/timing* guarantees ReactLynx code implicitly depends on (e.g., that a BTS-side synchronous UI query reflects the main thread's state as of the last flush, not a stale or a future one).
- Every message is one of `RpcMessageData` (`{name, data, sync:false, retId?, hasTransfer?}`) or `RpcMessageDataSync` (`{name, data, sync:true, lock, buf}`) — i.e., the "protocol" is not a stable set of message *shapes* per feature, it's a single generic envelope wrapping a named, statically-typed endpoint (`createRpcEndpoint` in `RpcEndpoint.ts`), so from lynx-vello's point of view the thing to port is the *endpoint catalog* (below), not a wire format.

### What actually triggers a render / patch

There is no continuous "diff" loop. The flow is pipeline-oriented and push-based:

1. **Initial load**: `LynxView.ts`'s `#render()` creates the iframe realm, then `TemplateManager.fetchBundle` (`.../mainthread/TemplateManager.ts`) streams decoded template sections in from a dedicated **third** worker (the decode worker, `.../decodeWorker/decode.worker.ts` — separate from both MTS and BTS, purely for off-main-thread binary parsing) and calls back into `LynxViewInstance` per section: `onPageConfigReady` (installs the Element PAPIs + main-thread globals onto the iframe's `globalThis` — see `createElementAPI.ts`/`createMainThreadGlobalAPIs.ts`), `onStyleInfoReady` (pushes the decoded `StyleSheetResource` into the Rust `MainThreadWasmContext.push_style_sheet`), `onMTSScriptsLoaded` (evaluates the mts/"lepus" root chunk **inside the iframe**, synchronously, via `mtsRealm.loadScript`), then `onBTSScriptsLoaded` (registers the bts chunk with the worker's chunk cache and calls `backgroundThread.startBTS()`).
2. **`onMTSScriptsExecuted`** (called right after the root mts chunk runs) calls `this.mainThreadGlobalThis.renderPage?.(processedData)` then **`this.mainThreadGlobalThis.__FlushElementTree()`** — this is the actual "commit" signal. `renderPage`/`__FlushElementTree` are functions the mts/lepus bundle defines and calls; `__FlushElementTree`'s *implementation*, provided by web-core (`createElementAPI.ts`'s returned object), is where the freshly-built `page` element is appended into the shadow root, `dispatch_start`/`layout_start`/`ui_operation_flush_start/end` timing marks are emitted, the lynx-view's `load` event fires, and per-element exposure (`uiappear`/`uidisappear`) status is recomputed and enabled/disabled events flushed.
3. **Subsequent updates**: BTS calls `updateData`/`updateGlobalProps`/`updateCardData` (React state changes, `lynx.updateGlobalProps`, `<lynx-view>.updateData()`), which RPC over to `LynxViewInstance.updateData`/`updateGlobalProps`; these in turn call the mts realm's `updatePage`/`processData` — again mts-defined functions — which mutate the DOM via the Element PAPIs and are expected to end with another `__FlushElementTree()` call (a "pipeline"; `pipelineOptions.pipelineID` threads through the whole timing chain visible in `markTiming` calls). **The main thread never diffs anything itself** — the ReactLynx runtime in the worker (packages/react/runtime) does the vdom/snapshot diffing and emits a sequence of granular element-API calls (`__CreateElement`, `__SetAttribute`, `__AppendElement`, `__SwapElement`, …) which mts executes as "Leo Assembly" opcodes (see `web-core/src/leo_asm` and `AGENTS.md`'s description) directly against the Rust `MainThreadWasmContext`, which owns a `unique_id -> WeakRef<HtmlElement>` map (`main_thread_context.rs`) and mutates the real DOM element-by-element as each opcode streams in — followed by exactly one `__FlushElementTree` per pipeline.
4. **Events flow the opposite direction.** A real DOM event (click, touch, scroll…) is captured by a single delegated listener the wasm core registers per-event-name the first time any element asks for it (`MainThreadWasmContext::enable_event` → `mts_binding.add_event_listener`), then dispatched by unique-id bubble path through `dispatch_event_by_path`/`common_event_handler` (`event_apis.rs`) — an explicit two-phase capture-then-bubble walk over `capture-bind`/`capture-catch`/`bindevent`/`catchevent`, matching Lynx's own event model (allowlisted "reactive" element events, e.g. list-refresh events, plus a `global-bindevent` fan-out independent of the DOM path). For each matching handler it calls back into JS via `RustMainthreadContextBinding` (`js_binding/mts_js_binding.rs`): `publish_event` (cross-thread — this posts an async RPC message, `publishEventEndpoint`/`publicComponentEventEndpoint`, into the BTS worker, where `tt`'s registered handler ultimately calls the React event handler) or `publish_mts_event` (a "worklet" handler — runs synchronously in-process inside the iframe realm, never crosses to BTS at all). This bind/catch/capture terminology and the worklet-vs-cross-thread branch is drawn straight from Lynx's native event model — a native lynx-vello implementation needs the same two-tier (in-thread worklet vs. cross-thread bind) event routing, not just "one event bus."

### Message/API catalog

All endpoints are declared once in `lynx-stack/packages/web-platform/web-core/ts/client/endpoints.ts` via `createRpcEndpoint(name, isSync, hasReturn, hasReturnTransfer?)`; every one of them is async in the current codebase (`isSync: false` for all), so the only calls that use the true blocking `Atomics.wait` path are ones that pass `isSync: true` to `createRpcEndpoint` directly — the only such caller found is the low-level primitive itself (`Rpc.invoke`'s sync branch), i.e. the framework exposes it as a mechanism but the current endpoint catalog doesn't exercise it (unlike native Lynx JSI, which does have true synchronous bridge calls) — **flag for lynx-vello**: don't assume the web target needs a synchronous cross-thread hop for anything currently listed here; treat "async round-trip" as the default and only special-case where genuinely needed (e.g. some `invokeUIMethod`/`getPathInfo`/`setNativeProps` callers expect a callback, implemented here as ordinary async promises, not the SharedArrayBuffer path).

### Notes for lynx-vello / W3C-divergence flags

- **Event model**: the capture/bubble split with allowlisted "reactive" events and a separate `global-bindevent` channel independent of the DOM tree path is Lynx-specific, not a W3C DOM event-model concept (W3C capture/bubble is purely tree-structural; there's no standard "global" event fan-out orthogonal to the tree). This is intentional Lynx behavior to replicate as-is (not a case where the brief's W3C-preference applies — it's an application-level pub/sub feature layered on top of, not replacing, standard dispatch).
- **Worklet vs. cross-thread event handlers** (`add_run_worklet_event` vs `add_cross_thread_event`, `publish_mts_event` vs `publish_event` in `event_apis.rs`) is the concrete mechanism lynx-vello must reproduce for its "gesture/worklet" story: worklet handlers must run synchronously in the same execution context as layout/DOM (mirrored by mts/iframe here), while ordinary bind handlers are always an async hop to the app-logic thread — this maps directly onto lynx-vello's native/main split even without literal OS threads.
- **Synchronous cross-thread calls via `Atomics.wait`/`SharedArrayBuffer`** are a real, intentional blocking primitive in the RPC layer (available, even if the current endpoint table doesn't use `isSync:true` anywhere) — flag this as a *browser-environment workaround*, not a behavior lynx-vello needs to reproduce mechanically: a native engine can make the equivalent calls (e.g. a background-thread synchronous UI query) trivially synchronous in-process. What must be preserved is the *ordering contract* — a "synchronous" query from application logic must observe exactly the main-thread state as of the most recent completed flush (not a torn/partial state mid-mutation), which is what the blocking wait effectively guarantees here by construction (the main thread can't be doing anything else while parked in `Atomics.wait`... except it isn't parked, the *caller* is; the main thread side runs the handler and replies — so the invariant is "handler runs to completion with a consistent, fully-flushed element tree," which lynx-vello should guarantee via its own scheduler instead of literal atomics).
- **No standard IPC framing to imitate**: because every endpoint shares one generic envelope (name + args + optional sync/return metadata), lynx-vello's job is to port the *endpoint catalog and its call/response semantics* (which calls are fire-and-forget vs. awaited, which cross a "thread" boundary vs. stay in-process/worklet), not the `postMessage`/`MessagePort` transport itself, which has no equivalent need in a single-process native engine — the transport should collapse to direct function calls or an internal task-queue, while the *endpoint semantics table above* becomes the contract lynx-vello's scheduler must uphold (e.g., still batch `markTiming` the way BTS does, still gate DOM mutation visibility on an explicit "flush" boundary matching `__FlushElementTree`, still keep worklet handlers off the async event queue).
- **`__FlushElementTree` as the sole commit boundary** is the key behavioral rule to carry over verbatim: lynx-vello's native renderer should likewise expose exactly one explicit "commit/flush" call that (a) is the only point at which pending element-tree mutations become visible/paint-eligible, (b) is where timing marks and exposure (`uiappear`/`uidisappear`) recomputation happen, and (c) is what unblocks the very first page attach (`page && !page.parentNode` gate in `createElementAPI.ts`). There is no implicit per-mutation reflow/rerender in this model — batching all mutations between flushes is load-bearing for both performance and (likely) for exact ReactLynx timing-hook compatibility.
