# DOM/element model, event system & gesture recognizers

> Research: multi-agent sweep over `lynx/` and `lynx-stack/` (see [AGENTS.md](../../AGENTS.md) for the reference-repo shorthand and the W3C-first standards policy). Supersedes the earlier stub.

### Event Model, bind/catch Dispatch & Gesture Recognizer System

**Scope confirmed by source.** The production dispatch engine is `lynx/core/renderer/events/touch_event_handler.{h,cc}` (81 KB, `TouchEventHandler`), backed by `lynx/core/renderer/events/events.h` (`EventHandler`, `EventOption`, `EventPhase`) and `lynx/core/renderer/events/event_handler.cc` (the `Is*Event()`/`GetEventPhase()` predicates). Gesture recognition is a newer, separate "Clay" UI subsystem at `lynx/clay/ui/gesture_handler/**` (2026-copyright files; arena/detector/handler/common subdirs) plus supporting velocity/mouse code in `lynx/clay/ui/gesture/**`; the `GestureType` enum contract is in `lynx/core/public/gesture_handler.h`. On the ReactLynx/web side, the identical event-type strings and gesture data model are re-implemented in `lynx-stack/packages/web-platform/web-core/ts/types/EventType.ts` and the standalone public API package `lynx-stack/packages/lynx/gesture-runtime/src/**`, consumed by `lynx-stack/packages/react/runtime/src/snapshot/gesture/processGesture.ts`. I could not find `NodesRef`/`SelectorQuery` node-tree query implementation details in this pass (only usages in `lynx-stack/packages/react/runtime/src/snapshot/lynx/nodesRef.ts`); that API is out of scope for this event/gesture section and not further researched here.

**The bind/catch/capture-bind/capture-catch model, precisely.** Lynx does not have a generic `addEventListener(type, handler, {capture})` API on elements at all — there is no element-level listener registration API in the runtime; bindings are static markup attributes compiled into the template's per-element event map (wire-format `type` string, see `docs/web-binary-template.md`). Each attached handler carries exactly one of five mutually-exclusive type tags (`lynx/core/renderer/events/events.h`: `kEventBindEvent="bindEvent"`, `kEventCatchEvent="catchEvent"`, `kEventCaptureBind="capture-bind"`, `kEventCaptureCatch="capture-catch"`, `kEventGlobalBind="global-bindEvent"`), and `EventHandler::GetEventPhase()` (`lynx/core/renderer/events/event_handler.cc:82-90`) maps bind/catch → `kBUBBLING_PHASE` and capture-bind/capture-catch → `kCAPTURING_PHASE` — i.e. the *phase a handler runs in is baked into which attribute name authored it*, not a runtime flag on a shared listener as in DOM (`addEventListener(type, fn, {capture: true})` toggles phase on the *same* API). Dispatch is a two-pass walk over a precomputed ancestor chain (`GenerateResponseChain`, `touch_event_handler.cc:721-820`; chain[0] = target, chain[last] = root, i.e. target→ancestor order — component-boundary crossing is gated by `EventOption.composed_`). `TouchEventHandler::StartEventCapture` (`touch_event_handler.cc:1690-1761`) walks the chain **reverse** (root→target = capture order): it collects every `capture-bind` handler on every node encountered, but as soon as it hits a `capture-catch` handler it records that node's op and **stops walking further toward the target** (so a capture-catch on an ancestor prevents inner nodes, including the target itself, from ever seeing capture or bubble handlers). `TouchEventHandler::StartEventBubble` (`touch_event_handler.cc:1763-1835`) then walks the chain **forward** (target→root = bubble order): collects every `bindEvent` handler per node, and stops as soon as it hits a `catchEvent` handler on some node (that node's catch handler fires, nothing above it runs). Firing itself happens in `StartEventFire` (`touch_event_handler.cc:1837-1935`), which supports `stopPropagation()`/`stopImmediatePropagation()` (exposed to Lepus/worklet code via `__StopPropagation`/`__StopImmediatePropagation` native functions, `lynx/core/runtime/lepus/bindings/renderer_functions.cc:1277-1300`) — these behave like DOM's, gating remaining ops in the *same* fired pass — but there is **no `preventDefault()`**: touch events are not cancelable, so a handler cannot suppress the runtime's own built-in behavior (e.g., scroll-view panning) via the event object; that is instead controlled out-of-band through the gesture arbitration API (`consumeGesture`/`interceptGesture`, see below). `global-bindEvent` is a third orthogonal channel — `HandleGlobalBindAndTriggerEvent` (`touch_event_handler.cc:1627-1688`) fires registered elements by event name across the whole page/component tree regardless of the response chain, driven by `ElementManager::GetGlobalBindElementIds`; it has no capture/bubble phase at all (`GetEventPhase()` returns `kNONE` for it). Additionally there is a hard-coded, non-standard coupling between two specific event names: `EVENT_TAP` is unconditionally suppressed if `EVENT_LONG_PRESS` was "consumed" (had any bound handler) earlier in the same touch sequence (`touch_event_handler.cc:1860-1868` and `:199-219`, tracked via `long_press_consumed_`) — this pairing of a discrete tap gesture with a timed long-press gesture, and the "loser is silently dropped" rule, has no DOM equivalent (DOM's `click` fires independent of any custom press-timer logic).

**What we should build instead (W3C-compliant target).** Implement standard DOM Level 3 event dispatch: a single capturing+bubbling walk per dispatched event (target chain computed once, ancestor→target for capture, target→ancestor for bubble, exactly as `GenerateResponseChain`'s ordering already gives us for free), where each listener is registered with an explicit `capture: bool` flag rather than encoded in a distinct attribute-name namespace. Concretely: parse `bindEvent`/`capture-bind` as `addEventListener(type, fn, {capture: type==='capture-bind'})`, and treat `catchEvent`/`capture-catch` as the same registration *plus* an implicit `stopPropagation()` call before invoking any following listener — this reproduces Lynx's exact "catch halts propagation" behavior while still allowing (unlike Lynx) multiple listeners at the same node/phase to coexist and firing order to be determined by registration order rather than a single privileged catch handler. Support real `Event.stopPropagation()`, `Event.stopImmediatePropagation()`, and add `Event.preventDefault()` + `Event.cancelable` (Lynx has neither concept for touch events — this is a **divergence to flag**: Lynx's touch/tap/longpress events cannot be canceled by application code, so gesture suppression must go through the separate arena/state-manager API described below; we should still expose standard `preventDefault()` on our synthetic touch events for web-platform-code compatibility, but our built-in recognizers (scroll, pan) should key off the Gesture API's `consumeGesture`/`interceptGesture`/`fail`/`end`, mirroring Lynx, not off `preventDefault()`). Drop the special-cased tap/longpress-suppression coupling as a global rule; instead let it fall out naturally of gesture-recognizer arbitration (a `LongPressGesture` and a `TapGesture` in a `waitFor`/race relationship) so apps that don't use the gesture API don't inherit an invisible cross-event suppression rule.

**Gesture recognizer API — registration, state machine, arbitration.** Registration is data-driven per-element: a `GestureDetector` (`lynx/clay/ui/gesture_handler/gesture_detector.h`, mirrored non-Clay in `lynx/core/renderer/events/gesture.h`'s `GestureDetectorImpl`) carries a `gesture_id`, a `GestureHandlerType`/`GestureType` (`PAN=0, FLING=1, DEFAULT=2, TAP=3, LONG_PRESS=4, ROTATION=5, PINCH=6, NATIVE=7`), a list of named worklet callbacks, a config map, and a `relation_map` with exactly three string-keyed relation lists: `"simultaneous"`, `"waitFor"`, `"continueWith"` (constants `kGestureSimultaneous`/`kGestureWaitFor`/`kGestureContinueWith` in `lynx/core/renderer/events/gesture.h:59-61`, and identically named on the TS side as `BaseGesture.simultaneousWith`/`waitFor`/`continueWith` in `lynx-stack/packages/lynx/gesture-runtime/src/gestureInterface.ts:16-24` and serialized in `lynx-stack/packages/lynx/gesture-runtime/src/baseGesture.ts:241-256`). The public JS-facing constructors live in `lynx-stack/packages/lynx/gesture-runtime/src/index.ts`: `PanGesture`, `TapGesture`, `LongPressGesture`, `FlingGesture`, `NativeGesture`, `DefaultScrollGesture`, plus composition helpers `Gesture.Simultaneous(...)`, `Gesture.Race(...)`, `Gesture.Exclusive(...)` (`composition.ts`). **`ROTATION` and `PINCH` are declared in every enum (`GestureType`/`GestureHandlerType`/`GestureTypeInner`) but I found no corresponding `*_gesture_handler.{h,cc}` implementation files under `lynx/clay/ui/gesture_handler/handler/` (only pan/tap/longpress/fling/native/default exist) — treat rotation/pinch as reserved/unimplemented in the current engine, not as a behavior we must match yet.** Each per-element gesture becomes a `BaseGestureHandler` subclass instance (`lynx/clay/ui/gesture_handler/handler/base_gesture_handler.h`) with an explicit state machine, `GestureConstants` (`base_gesture_handler.h:42-47`): `LYNX_STATE_INIT=0 → LYNX_STATE_BEGIN=1 → LYNX_STATE_ACTIVE=2 → LYNX_STATE_END=4`, with `LYNX_STATE_FAIL=3` reachable from BEGIN/ACTIVE, and `LYNX_STATE_UNDETERMINED=5` as a pre-BEGIN idle marker; corresponding lifecycle callbacks `onTouchesDown/onTouchesMove/onTouchesUp/onTouchesCancel` (raw pointer feed, always called) and `onBegin/onStart/onUpdate/onEnd` (state-transition callbacks; `onUpdate` only exists on `ContinuousGesture` subclasses per `lynx-stack/packages/lynx/gesture-runtime/src/baseGesture.ts:271-298`). From worklet code a `StateManager` (`gestureInterface.ts:160-166`, wired in `baseGesture.ts:46-106`) exposes `active()/fail()/end()` (force a state transition — these round-trip through `__SetGestureState` back into C++ `HandleGestureDetectorState`, `gesture_handler_trigger.cc:113-130`) and `consumeGesture(bool)`/`interceptGesture(bool)` (round-trip through `__ConsumeGesture`) — this is the mechanism an application uses to claim/veto a gesture instead of DOM's `preventDefault()`. **Arbitration** is a single-winner-plus-parallel-simultaneous-set model, not a full W3C-style independent-listener bubbling: `GestureArenaManager` (`lynx/clay/ui/gesture_handler/arena/gesture_arena_manager.{h,cc}`) builds a "compete chain" from the hit-tested response chain via `GestureDetectorManager::ConvertResponseChainToCompeteChain` (`lynx/clay/ui/gesture_handler/detector/gesture_detector_manager.cc:53-155`), which reorders/filters candidates according to each detector's `waitFor` (a gesture cannot win while a gesture it must wait for is still pending — it's skipped until that gesture resolves) and `continueWith` (chains handoff to a listed gesture after the current one ends) lists; `GestureHandlerTrigger::ReCompeteByGestures` (`gesture_handler_trigger.cc:258-373`) walks that chain on every touch move/up to find the next non-failed, non-ended candidate, and `GestureDetectorManager::HandleSimultaneousWinner` (`gesture_detector_manager.cc:157-216`) computes the parallel set of members that get **both** dispatched the same motion event as the sole `winner_` (`DispatchMotionEventWithSimultaneousAndReCompete`, `gesture_handler_trigger.cc:462-483`, `handle_by_simultaneous=true` flag) whenever a `"simultaneous"` relation lists them — everyone else not `simultaneous` with the winner is forced to `Fail()` (`FailOthersMembersInRaceRelation`, `gesture_handler_trigger.cc:183-202`), which is the "race" (exclusive-wins-unless-declared-simultaneous) semantics `Gesture.Race`/`Gesture.Exclusive` surface at the JS level. Fling has bespoke handling: a velocity threshold (`GestureConstants::FLING_SPEED_THRESHOLD = 300`, `base_gesture_handler.h:57`) captured at touch-up drives a `FlingScroller` (`lynx/clay/ui/gesture_handler/handler/fling_scroller.{h,cc}`) that synthesizes further motion callbacks after the finger lifts, re-running the same arbitration loop each tick (`GestureHandlerTrigger::FlingCallback`, `gesture_handler_trigger.cc:91-111`).

**W3C-standard equivalent to implement.** There is no W3C standard for exclusive multi-recognizer touch gesture arbitration (this is squarely native-UIKit/Android-GestureDetector territory, not a DOM concept), so this is a case where we should *port Lynx's actual behavior* (as the task brief anticipates for gesture semantics) rather than defer to a "more standard" alternative — ReactLynx app authors depend on the exact `waitFor`/`simultaneousWith`/`continueWith`/state-manager semantics. The one piece that should follow web platform convention instead is the underlying pointer feed: implement recognizers on top of standard Pointer Events (`pointerdown/pointermove/pointerup/pointercancel` with unified mouse+touch+pen semantics and pointer capture) rather than Lynx's touch-only `PointerEvent`/`TouchEvent` split, and expose `preventDefault()` on the low-level synthetic touch events (Lynx doesn't have it) purely for compatibility with any pre-existing DOM-shaped app code, while keeping gesture-level cancellation on the `fail()`/`consumeGesture()` API to match Lynx script behavior.

### Nested-Scroll Coordination Protocol (parent/child scroll gesture handoff)

Lynx implements a genuine, dedicated nested-scroll-parent/child cooperative protocol — architecturally analogous to Android's `NestedScrollingParent2`/`NestedScrollingChild2` — and it exists **twice**: once in the cross-platform C++ `clay` engine (`clay/ui/component/nested_scroll/*`, used by all of Android/iOS/Harmony via the shared renderer) and once again natively re-implemented on top of AndroidX's real `NestedScrollingChildHelper`/`NestedScrollingParentHelper` in Java/Kotlin Android UI views (`ui/scroll/NestedScrollView.java`, `ui/list/container/NestedScrollContainerView.java`, and the xelement `scroll-coordinator` family). iOS and Harmony have their own protocol/enum mirrors (`UIScrollView+Nested.h`, `LynxBaseScrollView+Nested.h`, `lynx_base_scroll_view_nested.h`) but Harmony's forwards straight into ArkUI's native `NODE_SCROLL_NESTED_SCROLL` node attribute rather than reimplementing dispatch logic itself — confirmed by reading `platform/harmony/lynx_harmony/src/main/cpp/ui/utils/base_scroll_container.cc`. This is **entirely distinct** from the gesture-arbitration/competing-recognizer system (which decides which single recognizer wins a touch) — nested-scroll coordination instead governs, once a scroll gesture recognizer has already won, how consumed/unconsumed scroll delta and fling velocity flow between an already-agreed chain of nested scrollables (list-inside-scroll-view, pull-to-refresh, tab-view swipe-inside-list, collapsing-header/"scroll-coordinator" patterns). The `clay` `NestedScrollManager` does call into the gesture arena (`gesture_manager()->ShouldInterceptGesture()`) to abort an in-flight nested-scroll chain when a competing recognizer (e.g. a JS `catch-touchmove` or native gesture) intercepts mid-drag — this is the one integration point between the two systems.

**Core protocol mechanics (from `clay/ui/component/nested_scroll/nested_scroll_manager.{h,cc}` and `nested_scrollable.{h,cc}`, cross-validated against `nested_scrollable_unittests.cc`):**
- A **scrollable chain** is constructed lazily on drag-start (`NestedScrollManager::DragStart` → `ConstructScrollableChain`) by walking from the innermost/beginning scrollable (`FindBeginningScrollable`) outward via `FindNextScrollable`, which prefers an **explicit** wiring (`prev-scrollable`/`next-scrollable` id attributes resolved via `FindScrollableById`, see `clay/ui/component/scroll_view.cc:939-946`) and falls back to DOM ancestry (`FindAncestorNestedDraggableView`).
- Chain construction **stops** when: direction mismatch between adjacent scrollables (vertical vs horizontal), or a scrollable has `enable-nested-scroll=false` (the default!) — confirmed by `NestedScrollableTest.Chain` unit test.
- Per-drag-frame dispatch (`NestedScrollManager::DispatchScroll` / `NestedScrollable::HandleNestedScroll`) walks the chain and, per scrollable, applies one of three **`NestedScrollMode`** policies settable independently for forward vs backward scroll direction via `scroll-forward-mode`/`scroll-backward-mode` attributes (parsed in `scroll_view.cc:ParseNestedScrollMode`): `self-only` (never hands off — outer never scrolls, confirmed test `ScrollMode`), `self-first` (default; inner consumes first, unconsumed passed to outer), `parent-first` (outer consumes first, remainder falls to inner).
- **Overscroll/rubber-band handoff**: if any scrollable in the chain is already mid-overscroll, that overscroll is resolved/restored *first*, before normal scroll dispatch resumes (test `RestoreOverscrollFirst`); overscroll on the current scrollable is deferred until *after* the whole chain (including parent) has been given a chance to consume the delta (`HandleNestedScroll`, "Let the overscroll effect be handled at last").
- **Fling handoff**: after drag-end, `NestedScrollManager::DragEnd` computes whether to `StartFling` (velocity threshold check `ShouldTriggerFling`, respecting overscroll boundary) or `TryStartBounce`. Fling deltas are re-dispatched through the *same* chain-walking `DispatchScroll` each animation tick (`OnDynamicAnimationUpdate`), so an outer scrollable picks up unconsumed fling velocity once the inner one hits its bound (test `NestedFling`) — and if the chain hits a hard boundary mid-fling, the fling is cancelled and a bounce is started on the scrollable that produced unconsumed delta.
- A separate **`RasterFlingManager`** (`raster_fling_manager.{h,cc}`) can offload fling animation to the compositor/raster thread for `ScrollView` only (not lists, not while under overscroll); when the raster-thread fling ends with remaining velocity, control is handed back to `NestedScrollManager::OnFlingEndFromRaster` to continue the nested-scroll chain on the UI thread — an extra thread-handoff dimension with no obvious equivalent needed in a single-threaded native Rust renderer, but the *behavioral contract* (remaining velocity continues the chain) must be preserved.
- Scroll status (`kIdle`/`kDragging`/`kFling`/`kBounce`) is itself handed off between scrollables in the chain as consumption shifts (`OnScrollableScroll` reassigns `current_scrollable_` and forwards status) — touch-slop is also globally reset to "resolve immediately" for the whole chain once any scrollable enters fling/bounce, to avoid arena delay/jank on the next drag.

**Explicit chain wiring vs. implicit DOM ancestry** is a key, easy-to-miss detail: `prev-scrollable`/`next-scrollable` string-id attributes let authors override the default parent/child relationship (e.g. skip an intermediate non-scrolling wrapper, or link two scrollables that aren't DOM ancestors at all) — a Rust rewrite must support id-based lookup (`FindScrollableById`), not just DOM-tree walking.

**Android-native duplicate protocol**: separately from `clay`, Android's Java/Kotlin UI layer re-implements real `NestedScrollingParent2`/`NestedScrollingChild2` (AndroidX interfaces: `onStartNestedScroll`/`onNestedScrollAccepted`/`onNestedPreScroll`/`onNestedScroll`/`onNestedPreFling`/`onNestedFling`/`onStopNestedScroll`) in `NestedScrollView.java` (a scroll-view acting as both parent and child) and `NestedScrollContainerView.java` (the list container, also both parent and child, plus item-snap paging support). This is the textbook Android nested-scroll dance: pre-scroll (parent gets first refusal) → self-scroll → post-scroll (unconsumed bubbles to parent) → pre-fling (parent gets first refusal on fling) → fling. The `lynx_xelement_scroll_coordinator` module (`ScrollCoordinatorLayout.kt`, `LynxUIScrollCoordinator.kt`) implements a Material-Design-style **CoordinatorLayout + AppBarLayout collapsing-header** pattern (`scroll-coordinator` tag) that is a `NestedScrollingParent` (for its internal AppBarLayout collapse) by default, but can *also* opt into being a `NestedScrollingChild` via the `android-nested-scroll-as-child` prop so it composes correctly when embedded inside another outer scroller — i.e., the same parent/child duality clay implements generically, reimplemented ad hoc for this one xelement.

**iOS and Harmony** mirror the same `NestedScrollMode` enum (with a 4th value, `NestedScrollModeParallel`, present in iOS/Harmony headers but **not** in clay's `NestedScrollMode` — worth flagging as a possible cross-platform inconsistency to resolve when designing the Rust model) via `LynxNestedScrollProtocol` (iOS, hit-test-chain based: `generateNestedScrollChainWithHitTestTarget:`) and `LynxBaseScrollViewNested` (Harmony, delegates dispatch to ArkUI's native nested-scroll node attribute rather than reimplementing it in C++).

**Web-target gap confirmed**: grepped `lynx-stack/packages/web-platform/web-elements/src/elements/ScrollView/*` and `XList/*` for `nested`, `enable-nested-scroll`, `scroll-forward-mode`, `scroll-backward-mode`, `prev-scrollable`, `next-scrollable`, `overscroll-behavior`, `touch-action` — **zero matches**. `ScrollView.ts`/`scroll-view.css` implement scrolling purely via native `overflow: scroll`/`overflow: clip` CSS with no JS-level nested-scroll delegation and no `overscroll-behavior` CSS property set at all, meaning nested `scroll-view`s on the web target rely entirely on the browser's default scroll-chaining, which bubbles overscroll to ancestors uncontrollably and has **no** self-only/self-first/parent-first distinction, no explicit chain wiring via ids, and no coordinated fling handoff. This is a real, unaddressed compatibility gap in the existing "Lynx for Web" reference architecture — lynx-vello should not copy this gap forward; a native engine has the opportunity (and, per the compatibility goal, the obligation) to implement the real protocol.

### Pointer-Events Control, Hit-Testing Overrides &amp; Touch Hit-Slop Expansion

Contrary to the initial "genuinely absent" hypothesis, Lynx **does** have a full family of hit-testing controls — they were simply invisible to a `css_property.h` grep because the CSS property tables in that header (and the `kPropertyID*` enum) are code-generated at build time from JSON specs in `lynx/tools/css_generator/css_defines/`, not hand-written. `pointer-events` is CSS-property id 225 (`lynx/tools/css_generator/css_defines/225-pointer-events.json`), and it carries an explicit, machine-readable compat matrix that says **`web_lynx: version_added: false`** — i.e. Lynx's own authors documented that this property is intentionally unimplemented on the web target, which is exactly the gap lynx-vello (consuming the same `.web.bundle` that web-core consumes) inherits unless it chooses to add support. Separately, there is a real per-element **hit-slop** attribute (`hit-slop`, non-CSS UI attribute, shipped since v2.14) present on iOS, Android and Harmony — this is the actual "expand/shrink touch target beyond visual bounds" mechanism the critic was looking for — plus a distinct global **touch-slop** (`tapSlop`/`touchSlop`, response-chain-only, default 8pt/dp) that is *not* per-element and is really a gesture-arbitration tolerance rather than a hit-slop. There is also a whole secondary "event-through" family (`event-through`, `event-through-active-regions`, `events-pass-through`, `block-native-event`, `block-native-event-areas`) that changes *which* element is considered hit or whether a hit is allowed to fall through to what's beneath/behind it, layered on top of (and evaluated after) `pointer-events`. None of this — not `pointer-events`, not `hit-slop`, not the event-through family — has any implementation in lynx-stack's web-platform; the only `pointer-events` occurrences there are hardcoded internal CSS in web-elements' own shadow-DOM templates (overlay component chrome), not a pass-through of the Lynx CSS property.

**Core hit-testing algorithm (`lynx/core/renderer/dom/fragment/event/platform_event_target.cc`, `PlatformEventTarget::HitTest`):** depth-first, reverse z-order (topmost/last child first) tree walk. For each candidate, `ShouldHitTest()` (= `user_interaction_enabled_`) gates whether it's considered at all; `ContainsPoint()` does an axis-aligned bounds check in the child's local coordinate space (no touch-slop expansion at this layer — slop is applied only in the platform UI layer's `containsPoint`, see below). Recurses into the first child whose bounds contain the point. After finding a candidate, if that candidate (or its resolved descendant) has `pointer-events: none`, the walk **backtracks to try the next sibling underneath** rather than falling through directly to the parent — this is the Lynx-specific pointer-events resolution order, implemented identically on iOS (`lynx/platform/darwin/ios/lynx/ui/LynxUI.m:3888-3910`, `hitTest:withEvent:`) and mirrored conceptually in the generic C++ path.

**Confirmed absences (searched, not found):**
- No `hit-slop` / `hitSlop` string anywhere in `lynx-stack/packages/web-platform/**` or `lynx-stack/packages/react/**` (web target has zero hit-slop support; native targets all have it).
- No CSS `pointer-events` implementation in `lynx-stack/packages/web-platform/web-core*`; the only occurrences in `lynx-stack/packages/web-platform/web-elements/src/template.rs` and `.../elements/htmlTemplates.ts` are hardcoded `pointer-events: none`/`auto` in the package's own internal overlay shadow-DOM CSS templates (component implementation detail), not a pass-through of a Lynx `pointer-events` style value. Combined with Lynx's own compat JSON marking `web_lynx: version_added: false`, this is very likely an *intentional* scope decision by the Lynx team (not an accidental gap), but it is still a gap lynx-vello inherits and should decide on explicitly.
- `css_property.h` itself contains no literal `PointerEvents`/hit-slop tokens because it (and the `kPropertyID*` enum) is generated at build time from `lynx/tools/css_generator/css_defines/*.json`; the source of truth for the CSS property is `lynx/tools/css_generator/css_defines/225-pointer-events.json`, not the checked-in generated header — worth noting for anyone repeating this kind of search in this codebase.
